# 嵌入式项目分层架构规范

## 1. 文档目的

本文档用于统一嵌入式固件项目的分层设计、目录组织、语言选择、依赖方向与编码约束。

目标不是追求形式上的整齐，而是解决以下问题：

- 降低维护成本
- 降低跨模块耦合
- 方便扩展新功能、新协议和新中间件
- 让多人协作时清楚代码应该写在哪一层
- 让项目在不引入 RTOS 的前提下也能保持结构清晰

本文档适用于基于 STM32 一类 MCU 的固件工程，尤其适合存在以下特征的项目：

- 使用 HAL / CubeMX
- 使用中断 + DMA + 主循环
- 存在多种工作模式
- 可能引入 FatFs、Modbus、自定义协议、Flash 存储等组件

## 2. 推荐目录结构

```text
Project/
├─ BSP/
│  ├─ uart/
│  ├─ rs485/
│  ├─ sdcard/
│  └─ flash/
│
├─ Middleware/
│  ├─ fatfs/
│  ├─ modbus/
│  └─ custom_protocol/
│
├─ Services/
│  ├─ storage/
│  ├─ protocol/
│  ├─ modbus/
│  └─ report/
│
├─ Domain/
│  ├─ measurement/
│  ├─ calibration/
│  ├─ filter/
│  └─ model/
│
├─ App/
│  ├─ mode_manager/
│  ├─ command/
│  └─ usecase/
│
└─ System/
```

如果项目规模继续扩大，建议增加：

```text
Common/
├─ types/
├─ error/
├─ config/
│  ├─ os_config.h      # OS 配置（时基、任务数等）
│  ├─ bsp_config.h     # BSP 硬件配置（串口、GPIO、SPI 等）
│  └─ app_config.h     # APP 功能配置（版本号、功能开关等）
└─ utils/
```

`Common` 用于存放跨层共享、但不带业务语义的公共内容。

### 配置集中管理规范

所有配置集中在 `Common/config/` 目录下，按文件职责分类：

| 配置文件 | 职责 | 使用者 |
|---------|------|--------|
| `os_config.h` | OS 相关配置（时基周期、最大任务数等） | 所有层 |
| `bsp_config.h` | 硬件配置（串口选择、GPIO 引脚、SPI/ADC 等） | BSP 层、依赖 BSP 的 Services |
| `app_config.h` | APP 功能配置（版本号、功能开关、宏定义等） | App 层 |

#### 使用原则

1. **集中管理**：改配置只去 `Common/config/` 找，不用逐个文件夹翻
2. **按文件分类**：硬件配置放 `bsp_config.h`，功能配置放 `app_config.h`
3. **逻辑约束**：`bsp_config.h` 仅供 BSP 和 Services 使用，Domain 层禁止 include

#### 示例

```c
// Common/config/bsp_config.h
#define LOG_HUART           huart1      // 日志输出串口
#define UPGRADE_HUART       huart2      // 升级通信串口
#define LED_GPIO_PORT       GPIOF       // LED 端口
#define LED_GPIO_PIN        GPIO_PIN_10 // LED 引脚
#define LED_ACTIVE_HIGH     true        // LED 高电平有效

// Common/config/app_config.h
#define APP_CURRENT_VERSION  202606082257ULL
#define ENABLE_DEBUG_PRINT   1
#define LOG_Printf(...)      LogService::Instance().Printf(__VA_ARGS__)
```

#### 好处

- 换板子或改硬件时，只改 `bsp_config.h` 一个文件
- 发布时关闭调试输出，只改 `app_config.h` 的开关
- 不用记住"这个配置在哪个层"

## 3. 整体分层原则

### 3.1 核心原则

分层不是按"代码是谁生成的"来分，而是按"职责"和"依赖方向"来分。

判断一个模块该放哪层，可以用一句话概括：

> 看它依赖谁，又为谁提供能力。

### 3.2 依赖方向

建议采用以下依赖方向：

```text
App -> Domain -> Services -> Middleware / BSP
System -> App / Services
BSP -> HAL / Drivers
```

更严格地说：

- `App` 可以依赖 `Domain`、`Services`、`System`
- `Services` 可以依赖 `Middleware`、`BSP`，必要时可依赖 `Domain` 的数据模型
- `Domain` 尽量只依赖 `Common`
- `Middleware` 可以依赖 `BSP`
- `BSP` 只依赖 HAL、LL、寄存器定义或板级资源

### 3.3 禁止的反向依赖

以下依赖应明确禁止：

- `BSP -> App`
- `Middleware -> App`
- `Domain -> BSP`
- `Domain -> Middleware`
- `System -> 业务实现`

---

## 4. 各层职责定义

## 4.1 BSP 层

### 定义

`BSP` 负责板级硬件访问和 MCU 外设封装。

### 典型内容

- UART、RS485、SPI、I2C、ADC、TIM、GPIO、DMA、IWDG
- SD 卡底层驱动
- 外部 Flash 底层驱动
- LED、蜂鸣器、继电器、电源使能等板级控制
- 与 CubeMX/HAL 对接的外设初始化和最小封装

### 这一层应该做的事

- 初始化硬件
- 读写外设
- 提供稳定、简单的底层接口

### 这一层不该做的事

- 不写模式切换逻辑
- 不写业务状态机
- 不直接处理 Modbus 功能码
- 不实现文件系统策略
- 不做功率/电阻等领域算法

### 推荐接口风格

```c
bool BspUart_Send(const uint8_t *data, uint16_t len);
bool BspFlash_Read(uint32_t addr, uint8_t *buf, uint32_t len);
bool BspFlash_Write(uint32_t addr, const uint8_t *buf, uint32_t len);
void BspRs485_SetDirTx(void);
void BspRs485_SetDirRx(void);
```

### 推荐语言

`C`

### 原因

- 与 HAL / CubeMX 风格一致
- 中断、启动文件、驱动层天然适合 C
- 接近硬件，更轻、更稳

---

## 4.2 Middleware 层

### 定义

`Middleware` 负责第三方组件、通用协议栈和标准中间件。

### 典型内容

- FatFs
- FreeModbus 或其他 Modbus 实现
- 自定义协议栈
- 后续可能新增的 USB、lwIP、MQTT、LittleFS、FreeRTOS 等

### 这一层应该做的事

- 提供通用能力
- 维护协议栈或中间件本身的完整性
- 尽量保持对原库的最小修改

### 这一层不该做的事

- 不直接处理产品模式切换
- 不直接写业务命令语义
- 不直接访问业务上下文
- 不直接决定保存配置、写日志、上报数据这些产品行为

### 推荐语言

保持原中间件语言。

例如：

- FatFs：`C`
- FreeModbus：`C`
- 自定义协议：可用 `C` 或 `C++`

### 原则

不要为了"统一语言"强行重写成熟第三方库。

---

## 4.3 Services 层

### 定义

`Services` 负责把底层能力和中间件能力翻译成项目可直接使用的服务。

这一层是最关键的隔离层，负责切断业务层对第三方库和底层硬件的直接依赖。

### 典型内容

- `storage/`：配置存储、日志存储、记录存储
- `protocol/`：自定义协议接入、协议应答策略、命令翻译
- `modbus/`：寄存器映射、Modbus 命令适配
- `report/`：上报打包、发送节流、报文构造
- 也可以扩展出：
  - `adc_service/`
  - `capture_service/`
  - `uart_service/`

### 这一层的本质

它是"翻译层"。

例如：

- FatFs 提供的是 `f_open / f_write / f_read`
- 服务层对应用提供的是 `ConfigStore_Save / LogStore_Append`

又比如：

- Modbus 中间件提供的是功能码和寄存器读写
- 服务层对应用提供的是 `AppCommand` 或寄存器语义映射

### 这一层不该做的事

- 不直接维护完整业务模式状态机
- 不直接塞满复杂领域算法
- 不直接操作具体业务全局变量

### 推荐语言

优先 `C++`，必要时 `C`

### 原因

- 适合封装状态和上下文
- 适合提供清晰的服务接口
- 方便把底层能力整理为稳定 API

### 推荐接口风格

```cpp
class ConfigStore {
public:
    bool Save(const AppConfig& cfg);
    bool Load(AppConfig& cfg);
};

class ProtocolService {
public:
    bool Input(const uint8_t* data, uint16_t len);
    bool BuildAck(const AppCommand& cmd, TxFrame& frame);
};
```

---

## 4.4 Domain 层

### 定义

`Domain` 负责领域模型、核心算法和业务数据处理，是项目的核心知识资产。

### 典型内容

- 测量算法
- 滤波算法
- 校准算法
- 功率计算
- 毫欧计算
- 波形分析
- 领域数据模型

### 这一层应该做的事

- 输入数据
- 计算结果
- 输出领域对象或领域值

### 这一层不该做的事

- 不直接发送串口
- 不直接操作文件系统
- 不直接访问 HAL
- 不直接切换模式

### 推荐语言

优先 `C++`

### 原因

- 适合组织复杂数据结构与算法
- 适合表达领域模型
- 相比纯 C，更利于维护复杂逻辑

### 推荐接口风格

```cpp
PowerResult PowerCalc_413_457(const AdcFrame& frame, const CaptureInfo& capture);
uint16_t WaveFilter_Limit(uint16_t* in, uint16_t len, uint16_t level, uint16_t n);
bool Calibration_Apply(PowerResult& result, const CalibrationTable& table);
```

---

## 4.5 App 层

### 定义

`App` 负责业务流程编排、模式管理、命令处理和用例执行。

### 典型内容

- `mode_manager/`
- `command/`
- `usecase/`

### 这一层应该做的事

- 响应"切到功率模式"
- 响应"收到完整采样帧"
- 决定"先滤波，再计算，再上报"
- 管理当前工作模式和上下文

### 这一层不该做的事

- 不直接调用 `f_open / f_write`
- 不直接解析 Modbus CRC
- 不直接读写寄存器或外设细节
- 不直接操纵 DMA 缓冲

### 推荐语言

`C++`

### 原因

- 模式管理、命令编排和用例组织天然适合 C++
- 便于使用对象、上下文和接口表达业务流程

### 推荐结构

```text
App/
├─ mode_manager/
│  ├─ mode_wait/
│  ├─ mode_power/
│  ├─ mode_mohms/
│  └─ mode_gsm/
├─ command/
│  ├─ command_dispatcher/
│  └─ command_handler/
└─ usecase/
   ├─ switch_mode/
   ├─ save_config/
   └─ report_measurement/
```

---

## 4.6 System 层

### 定义

`System` 负责系统级运行支撑，不承载具体业务。

### 典型内容

- 调度器
- 事件队列
- 软定时器
- 软件时基
- 公共断言
- 通用环形缓冲

### 这一层应该做的事

- 提供任务调度与事件分发基础设施
- 提供通用的时间片和 Tick 驱动
- 提供统一的系统级工具

### 这一层不该做的事

- 不写功率模式逻辑
- 不写协议命令业务判断
- 不写具体测量算法

### 推荐语言

优先 `C`

### 原因

- 更轻量
- 更稳定
- 更适合作为通用基础设施被各层复用

---

## 5. 各层语言建议总结

| 层 | 推荐语言 | 说明 |
|---|---|---|
| BSP | C | 硬件近、稳定、贴近 HAL |
| Middleware | 保持原库语言 | 不强行改写第三方组件 |
| Services | C++ 优先 | 适合做状态封装与能力翻译 |
| Domain | C++ 优先 | 适合组织复杂算法和数据模型 |
| App | C++ | 适合模式管理与业务编排 |
| System | C 优先 | 适合调度、事件、定时器等系统基础设施 |

### 额外约束

- 不使用异常
- 尽量不依赖 RTTI
- 谨慎使用动态内存
- 优先使用静态对象、固定容量缓冲区、显式上下文

---

## 6. 统一编码与设计约束

## 6.1 中断约束

中断只允许做三件事：

- 采集数据
- 更新最小状态
- 投递事件或置标志

中断中禁止：

- 跑完整业务流程
- 跑复杂协议解析
- 写文件系统
- 完整执行模式切换

推荐写法：

```c
void HAL_ADC_ConvCpltCallback(ADC_HandleTypeDef* hadc)
{
    CaptureService_OnAdcDone();
    EventQueue_Post(EVT_ADC_FRAME_READY);
}
```

---

## 6.2 全局变量约束

原则：

- 禁止跨层随意访问全局变量
- 全局变量必须收口到统一上下文对象

推荐做法：

```cpp
struct AppContext {
    WorkMode mode;
    CaptureInfo capture;
    PowerResult power;
    MohmsResult mohms;
    SystemStatus status;
};
```

不推荐做法：

- `systemConfig` 到处改
- `ADCFinish` 到处读写
- `pulseWidth`、`batv`、`resSum` 散落在多个模块里

---

## 6.3 第三方库使用约束

原则：

- 业务层不得直接依赖第三方库 API
- 必须通过 `Services` 层使用中间件能力

例如：

- `FatFs` 只能通过 `Services/storage`
- `Modbus` 只能通过 `Services/modbus`
- 自定义协议只能通过 `Services/protocol`

---

## 6.4 模式切换约束

所有模式切换必须统一入口。

推荐：

```cpp
ModeManager_RequestSwitch(MODE_POWER);
```

禁止：

- 协议层直接改 `workStatus`
- 服务层偷偷改模式
- 中断里直接切换模式

推荐模式管理流程：

```text
RequestSwitch -> ExitOldMode -> ClearOldState -> EnterNewMode -> RunNewMode
```

---

## 6.5 命名与接口约束

建议各层使用一致前缀，避免职责不清。

### 推荐前缀

- `BspUart_*`、`BspFlash_*`
- `StorageService_*`、`ProtocolService_*`
- `WaveFilter_*`、`PowerCalc_*`
- `ModeManager_*`、`AppCommand_*`
- `Os_*`、`EventQueue_*`

### 接口原则

- 对外接口少而稳
- 不暴露内部实现细节
- 参数尽量使用明确的上下文对象和结构体

---

## 6.6 当前 XY7320 APP 框架的落地约束

当前 `XY7320` APP 工程已经采用如下目录：

```text
XY7320/
├─ App/
├─ BSP/
├─ Common/
├─ Core/
├─ Domain/
├─ Drivers/
├─ Middleware/
├─ Services/
└─ System/
```

为了避免目录有了、边界却继续混乱，后续新增代码建议遵守以下约束。

### 1. `main.c` 只做系统装配

`Core/Src/main.c` 允许保留：

- HAL / CubeMX 初始化
- `OS_Init()`
- `BSP_TIM_OS_Init()`
- `App_Main_Init()`
- `App_Man_Start()`
- `OS_Start()`

不再建议把完整业务流程继续写回 `while(1)`。

### 2. 任务函数只做一步业务推进

当前已经存在：

```text
App/usecase/task_led.cpp
App/usecase/task_update.cpp
```

这类任务函数建议保持统一风格：

- 每次只执行一步逻辑
- 不写死循环
- 末尾通过 `OS_DelayMs()` 让出 CPU

推荐形式：

```cpp
extern "C" void Task_LED(void *arg)
{
    Led_service::Instance().Update();
    OS_DelayMs(10);
}
```

### 3. 中断里只收数据、置位、转交

以当前串口升级接收为例：

- `USART2_IRQHandler()` 属于 `Core/Src/stm32f4xx_it.c`
- `App_UpdateConfig_HandleUartIdleIrq()` 只负责在 IDLE 时停 DMA、取长度、拷贝缓冲、恢复接收
- 完整协议解析与业务动作放到任务上下文里处理

这条规则的目的，是避免中断里继续堆协议和升级业务。

### 4. BSP 层只碰硬件，不碰产品行为

例如当前：

- `BSP/gpio/bsp_gpio.c` 负责 GPIO 开关、翻转
- `BSP/tim/bsp_tim_os.c` 负责 TIM14 Tick 驱动

后续不建议在 `BSP` 中直接加入：

- 固件升级状态机
- LED 模式切换策略
- 业务命令解释

### 5. Services 层负责封装状态与能力

当前 `Services/led_service/led_service.cpp` 的组织方式是合理示例：

- 对外提供稳定接口
- 内部管理状态
- 底层通过 `BSP_GPIO_*` 驱动硬件

后续 `update` 相关逻辑建议也往这个方向演进，把"接收流程、协议处理、存储调用"逐步从 `Core/Src/app_update_config.c` 中拆出来。

### 6. 升级模块建议继续拆成三块职责

当前升级链路主要集中在：

```text
Core/Src/app_update_config.c
Core/Src/app_version_config.c
App/usecase/task_update.cpp
```

后续建议目标：

- `task_update.cpp`：只负责任务调度
- `update_service`：负责流程推进
- `update_protocol`：负责版本帧解析
- `version_store`：负责 A1 / A2 版本配置存储

也就是说，`App_UpdateConfig_Poll()` 可以先继续存在，但不要无限扩张。

### 7. APP 与 Bootloader 的共用代码要谨慎

当前版本配置模块已经从根目录 `Common/app_version_config.*` 拆到 APP 与 Bootloader 各自工程内部。

后续原则：

- 可以共用协议约定
- 可以共用数据格式定义
- 但不要轻易让 APP 和 Bootloader 强依赖同一个源码实现文件

因为两边生命周期、职责和后续演进方向并不一致。

## 7. FatFs、Modbus、CubeMX 这类边界问题的处理方式

## 7.1 CubeMX 生成内容的判断原则

不要按"是不是 CubeMX 生成"来判断属于哪层。

要按职责判断。

例如：

- `gpio.c`、`usart.c`、`adc.c`：属于 `BSP`
- `fatfs.c`、`ff.c`、`diskio.c`：属于 `Middleware`
- `cmsis_os.c` 或调度 glue：属于 `System`

### 结论

> 工具生成位置不等于架构归属。

---

## 7.2 FatFs 的建议归属

即使 FatFs 是 CubeMX 集成的，也仍然属于 `Middleware`。

### 典型角色划分

#### BSP

- `sdcard`、`flash`、`spi` 等底层驱动

#### Middleware

- `ff.c`
- `ff.h`
- `diskio.c`
- `fatfs.c`
- `sd_diskio.c`

#### Services

- `ConfigStore`
- `LogStore`
- `RecordStore`

### 推荐调用关系

```text
App -> Services/storage -> Middleware/fatfs -> BSP/sdcard or BSP/flash
```

### 禁止做法

应用层直接散落调用：

```c
f_open(...)
f_write(...)
f_close(...)
```

### 推荐做法

应用层调用：

```cpp
ConfigStore_Save(cfg);
LogStore_Append(text);
```

---

## 7.3 Modbus 的建议归属

Modbus 协议栈属于 `Middleware`，其上必须再包一层 `Services/modbus`。

### 推荐分法

#### BSP

- UART / RS485 驱动
- DE/RE 控制

#### Middleware

- Modbus RTU/ASCII/TCP 协议栈
- CRC
- 功能码解析框架

#### Services

- 寄存器映射
- 协议到业务命令的翻译

#### App

- 模式切换
- 参数应用
- 状态上报

### 推荐调用关系

```text
App <- Services/modbus <- Middleware/modbus <- BSP/rs485
```

### 核心原则

协议层不直接改业务状态。

例如，不推荐在 Modbus 回调里直接写：

```c
systemConfig.workStatus = POWER_MODEL;
```

而应该翻译成统一命令，再交给应用层：

```cpp
AppCommand cmd = AppCommand::SwitchToPowerMode;
ModeManager_HandleCommand(cmd);
```

---

## 7.4 自定义协议的建议归属

自定义协议的编解码可放在 `Middleware/custom_protocol`，其对接业务的入口放在 `Services/protocol`。

推荐拆分：

- `Middleware/custom_protocol`
  - 帧解析
  - 编码
  - CRC
  - 转义
- `Services/protocol`
  - 命令翻译
  - 应答策略
  - 业务命令生成

这样以后即使同时支持：

- 自定义协议
- Modbus
- CAN 命令
- 蓝牙串口命令

也都可以统一翻译为 `AppCommand`，交给同一个应用层处理。

## 8. 推荐的任务调度与运行方式

当前这类项目不一定要直接上 RTOS，更适合采用：

> 事件驱动 + 轻量协作式调度器 + 明确状态机

### 推荐思路

- 中断只投递事件
- `System` 层提供调度与事件分发
- `App` 层消费事件并驱动模式状态机

### 推荐事件示例

- `EVT_ADC_FRAME_READY`
- `EVT_UART_PACKET_READY`
- `EVT_CAPTURE_READY`
- `EVT_KEY_EVENT`
- `EVT_MODE_CHANGE`

### 推荐任务示例

- `Task_EventDispatch`
- `Task_ModeRun`
- `Task_Report`
- `Task_Button`
- `Task_Health`

### 不推荐的结构

- 所有逻辑堆进一个 `while(1)`
- `if/else if(workStatus)` 里混协议、滤波、上报、硬件切换

## 9. 现有 STM32 项目改造建议

如果项目已经存在，不建议一次性重构全部层。

建议分三步：

### 第一步：先划清逻辑层次

- 明确哪些文件逻辑上属于 `BSP`
- 明确哪些文件逻辑上属于 `Middleware`
- 明确哪些业务逻辑应该进入 `Services` / `App`

即使物理目录暂时不搬，也先在文档和代码注释中写清逻辑归属。

### 第二步：先增加 Services 层

优先把以下直接依赖切开：

- `FatFs -> storage service`
- `Modbus -> modbus service`
- `自定义协议 -> protocol service`

### 第三步：再拆 App 层和调度层

- 增加 `ModeManager`
- 把大循环里的模式分支拆成独立模块
- 引入事件队列和协作式调度

## 10. 硬规则建议

如果要把本文档作为团队统一架构规范，建议至少落下这 6 条硬规则：

### 规则 1

业务代码不得直接调用第三方中间件 API。

### 规则 2

模式切换只能通过 `ModeManager` 统一入口。

### 规则 3

中断中不得执行完整业务流程。

### 规则 4

`Domain` 层不得依赖 `BSP` 或 `Middleware`。

### 规则 5

新增协议、存储或通信方式时，优先新增 `Services` 适配层，而不是直接修改 `App`。

### 规则 6

任何跨模块共享状态，必须进入统一上下文对象，不允许自由散布全局变量。

---

## 11. 一句话总结

这套架构可以概括为：

> 以 `BSP` 作为硬件基础，以 `Middleware` 承载第三方组件，以 `Services` 隔离外部依赖，以 `Domain` 承载核心算法与模型，以 `App` 组织业务流程，以 `System` 提供运行支撑。

它的关键不在于目录名字，而在于以下 4 点是否真正做到：

- 分层明确
- 依赖单向
- 中间件隔离
- 业务统一入口

如果这 4 点成立，即使物理目录暂时不完全理想，项目依然可以保持较好的可维护性。

## 12. 下一步落地建议

建议按如下顺序动项目：

1. 先标注现有文件的逻辑归属
2. 先补 `Services` 层，切断业务对中间件的直接依赖
3. 再引入 `ModeManager`
4. 最后引入 `System` 层的调度和事件分发

如果已经准备开始改现有工程，建议优先改这三块：

- 协议层与业务层解耦
- 采样/测量逻辑与主循环解耦
- 模式切换入口统一
