# 命名与注释规范

## 1. 文档目的

本文档统一 XY7320 项目的命名风格与注释风格，解决以下问题：

- BSP 层三种命名风格并存（`BSP_GPIO_*` / `Bsp_UartRcv_*` / `BspFlash_*`）
- Services 层类名不一致（`Led_service` / `update_service` / `version_store`）
- 注释详略不一，有的文件没有注释，有的注释过于冗长
- C 与 C++ 混用场景下缺乏统一约定

本文档配合《嵌入式项目分层架构规范》使用。

---

## 2. 命名规范总览

### 2.1 选定风格

本项目采用 **PascalCase 模块前缀 + 下划线** 风格。

选择理由：

- 与架构规范中推荐的 `BspFlash_*`、`VersionFrame_*` 一致
- 在嵌入式 C 项目中辨识度高，一眼能看出函数归属哪个模块
- C 和 C++ 两套规则互不冲突，`extern "C"` 桥接自然

### 2.2 风格速查表

| 类别 | 风格 | 示例 |
|------|------|------|
| C 函数 | `模块前缀_动作` | `BspFlash_Read()`、`OS_CreateTask()` |
| C 类型 (typedef struct/enum) | `模块前缀_名称_t` | `BspGpio_t`、`OS_TCB_t`、`VersionFrame` |
| C 宏 / 常量 | `模块前缀_名称` (全大写) | `VERSION_FRAME_ACK`、`OS_INVALID_TASK` |
| C 枚举值 | `模块前缀_名称` (全大写) | `OS_TASK_READY`、`OS_TASK_DELAYED` |
| C 文件名 | `snake_case` | `bsp_flash.c`、`version_frame.c` |
| C++ 类名 | `PascalCase` | `LedService`、`VersionStore` |
| C++ 方法名 | `PascalCase` | `Init()`、`Update()`、`ProcessFrame()` |
| C++ 成员变量 | `m_camelCase` | `m_gpio`、`m_mode`、`m_rxBuf` |
| C++ 文件名 | `snake_case` | `led_service.cpp`、`update_service.cpp` |
| 局部变量 | `camelCase` | `frameLen`、`isValid` |
| 函数参数 | `camelCase` | `*data`、`uint16_t len` |

## 3. C 代码命名规则

### 3.1 模块前缀定义

每个 C 模块必须有唯一的前缀，前缀由 **模块所属层缩写 + 模块名** 组成。

| 层 | 前缀格式 | 示例 |
|----|----------|------|
| BSP | `Bsp` + 模块名 | `BspGpio`、`BspUartRcv`、`BspFlash`、`BspTimOs` |
| System | `OS` 或系统模块名 | `OS`、`EventQueue` |
| Domain | 领域模块名 | `VersionFrame`、`WaveFilter`、`PowerCalc` |

### 3.2 函数命名

格式：`前缀_动词[名词]`

```c
/* BSP 层 */
void BspGpio_Init(BspGpio_t *dev);
void BspGpio_On(BspGpio_t *dev);
void BspGpio_Off(BspGpio_t *dev);
void BspGpio_Toggle(BspGpio_t *dev);
bool BspGpio_IsOn(BspGpio_t *dev);

void BspUartRcv_Init(UART_HandleTypeDef *huart, uint8_t *buf, uint16_t size);
void BspUartRcv_Start(void);
bool BspUartRcv_IsFrameReady(void);
uint16_t BspUartRcv_GetFrameLength(void);
void BspUartRcv_CopyFrame(uint8_t *dst);
void BspUartRcv_ClearFlag(void);
void BspUartRcv_SendAck(const uint8_t *data, uint16_t len);
void BspUartRcv_HandleIdleIrq(UART_HandleTypeDef *huart);

int BspFlash_Read(uint32_t addr, uint8_t *buf, uint32_t len);
int BspFlash_Write(uint32_t addr, const uint8_t *buf, uint32_t len);
int BspFlash_EraseSector(uint8_t sector);

/* System 层 */
void OS_Init(void);
void OS_Start(void);
void OS_Tick(void);
int32_t OS_CreateTask(void (*entry)(void *), void *arg, uint8_t priority);
void OS_DelayMs(uint32_t ms);

/* Domain 层 */
int VersionFrame_Parse(const uint8_t *data, size_t len, VersionFrame *out);
```

**动词选择约定：**

| 动词 | 含义 | 使用场景 |
|------|------|----------|
| `Init` | 初始化 | 硬件/模块初始化 |
| `Start` / `Stop` | 启动/停止 | 使能/禁用外设或功能 |
| `Read` / `Write` | 读/写 | 数据读写 |
| `Get` / `Set` | 获取/设置 | 属性或状态存取 |
| `Is` / `Has` | 判断 | 返回 bool 的状态查询 |
| `Create` / `Delete` | 创建/销毁 | 资源生命周期管理 |
| `Enable` / `Disable` | 使能/禁用 | 功能开关 |
| `Handle` | 处理 | 中断/事件处理入口 |
| `Parse` | 解析 | 协议/数据解析 |
| `Send` / `Recv` | 发送/接收 | 通信操作 |
| `On` / `Off` / `Toggle` | 开/关/翻转 | GPIO 类控制 |

### 3.3 类型命名

结构体用 `_t` 后缀（表示 typedef），枚举可以省略：

```c
/* 结构体 */
typedef struct {
    GPIO_TypeDef *port;
    uint16_t pin;
    bool active_high;
} BspGpio_t;

typedef struct {
    uint64_t version;
    uint8_t  flag;
} VersionFrame;

/* 枚举 */
typedef enum {
    OS_TASK_UNUSED    = 0,
    OS_TASK_READY     = 1,
    OS_TASK_RUNNING   = 2,
    OS_TASK_SUSPENDED = 3,
    OS_TASK_DELAYED   = 4
} OS_TaskState_t;
```

### 3.4 宏与常量命名

全大写 + 下划线分隔，必须带模块前缀：

```c
/* 配置宏 */
#define OS_TICK_MS          1
#define OS_MAX_TASKS        10

/* 协议常量 */
#define VERSION_FRAME_FLAG_NEED_DOWNLOAD  0x00U
#define VERSION_FRAME_FLAG_DOWNLOADED     0x01U
#define VERSION_FRAME_ACK                 "XYA1"

/* 错误码 */
#define OS_INVALID_TASK     (-1)
```

**头文件保护宏：**

格式：`XY7320_模块名_H`

```c
#ifndef XY7320_BSP_FLASH_H
#define XY7320_BSP_FLASH_H
/* ... */
#endif
```

## 4. C++ 代码命名规则

### 4.1 类名

PascalCase，不加下划线，不加 `_t` 后缀：

```cpp
/* ✅ 正确 */
class LedService { ... };
class UpdateService { ... };
class VersionStore { ... };

/* ❌ 错误（当前代码中的旧风格，需逐步迁移） */
class Led_service { ... };
class update_service { ... };
class version_store { ... };
```

### 4.2 方法名

PascalCase，与 C 函数风格统一：

```cpp
class LedService {
public:
    static LedService& Instance();
    void Init();
    void SetMode(Mode mode);     // ✅ 不写下划线
    Mode GetMode() const;        // ✅ 不写下划线
    void Update();
private:
    void TurnOn();
    void TurnOff();
    void UpdateBlink();
};
```

### 4.3 成员变量

`m_` 前缀 + camelCase：

```cpp
class LedService {
private:
    BspGpio_t  m_gpio;           // ✅
    Mode       m_mode;           // ✅
    uint32_t   m_lastToggleTick; // ✅
    uint32_t   m_blinkPeriodMs;  // ✅
    bool       m_state;          // ✅
};
```

### 4.4 枚举类

```cpp
class LedService {
public:
    enum class Mode {
        Off,       // PascalCase（enum class 有作用域，不需要前缀）
        On,
        Blink,
        Breathe
    };
};
```

如果使用传统 enum（无 class），则需要前缀：

```c
typedef enum {
    LED_MODE_OFF,
    LED_MODE_ON,
    LED_MODE_BLINK,
    LED_MODE_BREATHE
} LedMode;
```

### 4.5 单例模式统一写法

项目中所有 Services 层单例统一用以下模式：

```cpp
class XxxService {
public:
    static XxxService& Instance();
    void Init();
    void Update();
private:
    XxxService() = default;
};
```

`Instance()` 实现：

```cpp
XxxService& XxxService::Instance()
{
    static XxxService instance;
    return instance;
}
```

| BSP | `bsp_gpio.c` / `bsp_gpio.h` |

目录名统一使用 `snake_case`，与模块名对应：

```text
├── gpio/

Services/
├── led_service/

```

## 7. 注释规范

### 7.1 注释风格选择

本项目采用 **Doxygen 兼容注释**。

选择理由：

- CLion / VSCode 等 IDE 原生支持 Doxygen 提示和悬浮显示
- 未来可一键生成 API 文档
- 语法轻量，不增加太多书写负担

### 7.2 文件头注释

每个源文件必须有文件头注释，说明文件职责：

```c
/**
 * @file    bsp_flash.c
 * @brief   内部 Flash 读写驱动
 *          基于 HAL_FLASH，提供按字编程、扇区擦除接口
 */
```

不需要写作者、日期等信息（Git 已经记录了）。如果觉得 `@file` 和 `@brief` 太正式，最简形式：

```c
/**
 * 内部 Flash 读写驱动
 * 基于 HAL_FLASH，提供按字编程、扇区擦除接口
 */
```

### 7.3 头文件中的接口注释

**所有对外暴露的函数、类型、宏，在头文件中必须有注释。**

#### 函数注释

```c
/**
 * @brief  读取 Flash 数据（内部直接 memcpy）
 * @param  addr  起始地址（必须是 Flash 有效地址）
 * @param  buf   输出缓冲区
 * @param  len   读取字节数
 * @return 0 成功, -1 失败
 */
int BspFlash_Read(uint32_t addr, uint8_t *buf, uint32_t len);
```

**何时用完整格式（@brief + @param + @return）：**

- 参数含义不显然时（比如 `addr` 有对齐要求）
- 返回值有多种含义时（0 成功 / -1 失败）
- 函数行为有前置条件时

**何时用简略格式（只写 @brief 或行尾注释）：**

- 函数名已经能说明一切时
- 参数含义显而易见时

```c
/** 初始化串口 DMA 接收 */
void BspUartRcv_Init(UART_HandleTypeDef *huart, uint8_t *buf, uint16_t size);

/** 启动 DMA 接收 + 使能 IDLE 中断 */
void BspUartRcv_Start(void);

/** 是否有完整帧到达 */
bool BspUartRcv_IsFrameReady(void);
```

#### 类型注释

```c
/**
 * GPIO 设备句柄
 *
 * 封装一个 GPIO 引脚的端口、引脚号和有效电平。
 * 调用 BspGpio_Init() 前必须填充好这三个字段。
 */
typedef struct {
    GPIO_TypeDef *port;       ///< GPIO 端口（如 GPIOF）
    uint16_t     pin;         ///< 引脚号（如 GPIO_PIN_10）
    bool         active_high; ///< true=高电平有效, false=低电平有效
} BspGpio_t;
```

#### 宏 / 常量注释

```c
/** 版本帧标志位：需要下载升级 */
#define VERSION_FRAME_FLAG_NEED_DOWNLOAD  0x00U

/** 版本帧标志位：已下载完成 */
#define VERSION_FRAME_FLAG_DOWNLOADED     0x01U

/** ACK 应答码（APP 收到版本帧后返回，然后复位进入 Bootloader） */
#define VERSION_FRAME_ACK  "XYA1"
```

### 7.4 源文件中的实现注释

**函数实现处不需要重复头文件的注释。** 只在以下情况添加注释：

- 实现中有不显然的逻辑（"为什么这样做"）
- 有硬件相关的 workaround
- 有性能或安全相关的考量

```c
/**
 * 直接发送应答数据（绕过 HAL 状态机）
 *
 * 原因：DMA 接收模式下 HAL_UART 处于 Busy 状态，
 * 正常 HAL_UART_Transmit() 会返回 BUSY 错误。
 * 这里通过寄存器直接写入，先停 DMA 再发送。
 */
void BspUartRcv_SendAckDirect(const uint8_t *data, uint16_t len)
{
    /* 先停止 RX DMA，避免发送时 DMA 继续搬运 */
    if (uart == USART2) {
        DMA1_Stream5->CR &= ~DMA_SxCR_EN;
        while (DMA1_Stream5->CR & DMA_SxCR_EN);
    }
    /* ... */
}
```

### 7.5 C++ 类注释

```cpp
/**
 * 升级流程编排
 *
 * 职责：
 * 1. 初始化串口 DMA 接收
 * 2. 周期轮询：检查是否有新版本帧到达
 * 3. 解析版本帧 → 写 A2 → 发 ACK → 复位
 *
 * 调用链：
 *   Task_UpdateConfig → UpdateService::Update()
 *                          ├── BspUartRcv_*       (BSP: 串口收发)
 *                          ├── VersionFrame_Parse  (Domain: 协议解析)
 *                          └── VersionStore::*     (Services: 版本存储)
 *
 * @note 单例模式，通过 Instance() 获取实例
 */
class UpdateService {
public:
    static UpdateService& Instance();

    /** 初始化升级服务（初始化串口、写 A1 为当前版本） */
    void Init();

    /**
     * 周期调用
     * 在 Task_UpdateConfig 中每 10ms 调用一次，
     * 检查是否有新帧到达，有则解析并处理
     */
    void Update();

private:
    UpdateService() = default;
    uint8_t m_rxBuf[64]{};  ///< 接收缓冲区

    /** 处理一帧数据，返回 true 表示成功解析并触发升级流程 */
    bool ProcessFrame(const uint8_t *data, uint16_t len);
};
```

### 7.6 行内注释

用于解释"为什么"，而不是"做了什么"：

```c
/* ✅ 好的行内注释 — 解释原因 */
/* 版本未变则跳过写入，保护 Flash 寿命（约 10000 次擦写） */
if (config.a1.version == version && config.a1.flag == flag) {
    return true;
}

/* ❌ 差的行内注释 — 重复代码 */
/* 把 len 加 1 */
len = len + 1;
```

### 7.7 TODO / FIXME / NOTE 标记

用于标记待办、已知问题或需要注意的地方：

```c
/* TODO: 增加帧长度越界检查，当前 s_proc_buf 固定 64 字节 */

/* FIXME: SendAckDirect 硬编码了 DMA1_Stream5，只支持 USART2 */

/* NOTE: 此处必须先停 DMA 再发送，否则 DMA 和 UART 发送会冲突 */
```

**使用约定：**

| 标记 | 含义 | 要求 |
|------|------|------|
| `TODO:` | 待实现的功能或优化 | 附带简要说明 |
| `FIXME:` | 已知问题，需要修复 | 附带问题描述 |
| `NOTE:` | 需要注意的点 | 说明原因 |
| `HACK:` | 临时方案，不够优雅但能工作 | 说明为什么临时 |

---

## 8. 注释详略原则

不是所有代码都需要同样详细的注释。按模块特性决定详略：

| 模块特性 | 注释要求 | 原因 |
|----------|----------|------|
| 对外 API（头文件中的声明） | **必须注释** | 这是别人使用你模块的唯一参考 |
| 协议格式 / 数据帧定义 | **必须注释** | 协议细节容易遗忘，注释即文档 |
| 硬件 workaround | **必须注释** | 不写清楚，下次自己也会忘 |
| 函数名已经说明一切的简单函数 | 一行 `/** 简要说明 */` 即可 | 不要为了注释而注释 |
| 局部变量 / 显而易见的逻辑 | **不注释** | 代码本身就是注释 |
| CubeMX 生成的代码 | **不注释** | 改了也会被覆盖 |

## 9. 完整示例

### 9.1 C 模块头文件示例

```c
/**
 * @file    bsp_uart_rcv.h
 * @brief   串口 DMA 接收模块
 *          基于 DMA + IDLE 中断实现不定长帧接收，双缓冲机制
 */

#ifndef XY7320_BSP_UART_RCV_H
#define XY7320_BSP_UART_RCV_H

#include "main.h"
#include <stdint.h>
#include <stdbool.h>

#ifdef __cplusplus
extern "C" {
#endif

/**
 * @brief  初始化串口 DMA 接收
 * @param  huart  串口句柄（由 CubeMX 生成，如 &huart2）
 * @param  buf    DMA 接收缓冲区（调用方分配）
 * @param  size   缓冲区大小（字节）
 */
void BspUartRcv_Init(UART_HandleTypeDef *huart, uint8_t *buf, uint16_t size);

/** 启动 DMA 接收 + 使能 IDLE 中断 */
void BspUartRcv_Start(void);

/** 是否有完整帧到达（IDLE 触发） */
bool BspUartRcv_IsFrameReady(void);

/** 获取当前帧长度 */
uint16_t BspUartRcv_GetFrameLength(void);

/**
 * @brief  将帧数据拷贝到外部缓冲区
 * @param  dst  目标缓冲区（调用方确保容量 >= 帧长度）
 */
void BspUartRcv_CopyFrame(uint8_t *dst);

/** 清除帧就绪标志（必须在取走帧数据后调用） */
void BspUartRcv_ClearFlag(void);

/**
 * @brief  发送应答数据（阻塞）
 * @param  data  应答数据
 * @param  len   数据长度
 */
void BspUartRcv_SendAck(const uint8_t *data, uint16_t len);

/**
 * @brief  直接发送应答（绕过 HAL 状态机）
 *
 * 用于 DMA 接收模式下需要发送数据的场景。
 * 内部会停止 DMA、禁用中断，然后用寄存器直接发送。
 *
 * @param  data  应答数据
 * @param  len   数据长度
 */
void BspUartRcv_SendAckDirect(const uint8_t *data, uint16_t len);

/**
 * USART IDLE 中断处理入口
 * 在 stm32f4xx_it.c 的 USART2_IRQHandler 中调用
 */
void BspUartRcv_HandleIdleIrq(UART_HandleTypeDef *huart);

#ifdef __cplusplus
}
#endif

#endif /* XY7320_BSP_UART_RCV_H */
```

### 9.2 C++ 类头文件示例

```cpp
/**
 * @file    version_store.h
 * @brief   版本配置存储
 *          管理 Flash 中的 A1（当前版本）/ A2（目标版本）槽位
 */

#ifndef XY7320_VERSION_STORE_H
#define XY7320_VERSION_STORE_H

#include "version_frame.h"
#include <cstdint>

/**
 * 版本配置存储
 *
 * Flash 布局（Sector 11，地址 0x080E0000）：
 *   偏移 0x00: A1 槽位（32 字节）— 当前运行版本
 *   偏移 0x40: A2 槽位（32 字节）— 目标升级版本
 *
 * @note 单例模式，通过 Instance() 获取实例
 */
class VersionStore {
public:
    /** 版本槽位结构（与 Flash 中布局一致，32 字节） */
    struct Slot {
        uint32_t magic;       ///< 魔数，标识槽位有效
        uint64_t version;     ///< 版本号
        uint8_t  flag;        ///< 标志位
        uint8_t  reserved[7]; ///< 保留
        uint32_t tail;        ///< 尾标记
    };

    /** 版本配置（A1 + A2） */
    struct Config {
        Slot a1;
        Slot a2;
    };

    static VersionStore& Instance();

    /** 读取完整配置 */
    bool Read(Config &config);

    /**
     * @brief  更新 A1 槽位（当前运行版本）
     * @param  version  版本号
     * @param  flag     标志位
     * @return true 写入成功, false 写入失败
     */
    bool WriteA1(uint64_t version, uint8_t flag);

    /**
     * @brief  更新 A2 槽位（目标升级版本）
     * @param  version  版本号
     * @param  flag     标志位
     * @return true 写入成功, false 写入失败
     */
    bool WriteA2(uint64_t version, uint8_t flag);

    /** 判断是否需要进入升级模式（A2 有效且版本与 A1 不同） */
    bool ShouldEnterUpgrade();

    /** 判断槽位数据是否有效 */
    bool IsSlotValid(const Slot &slot);

private:
    VersionStore() = default;

    /** 构造一个有效槽位 */
    void MakeSlot(Slot &slot, uint64_t version, uint8_t flag);

    /**
     * 保存配置到 Flash
     * NOTE: 每次调用会擦除整个 Sector 11（128KB），然后写回 A1+A2
     */
    bool Save(const Config &config);
};

#endif /* XY7320_VERSION_STORE_H */
```

## 10. 硬规则

以下规则在代码审查中必须严格执行：

### 命名硬规则

1. **C 函数必须有模块前缀**，不允许裸函数名（如 `Init()` → `BspFlash_Init()`）
2. **C++ 类名必须 PascalCase**，不允许下划线（如 `LedService`，不是 `Led_service`）
3. **C++ 方法名必须 PascalCase**，不允许下划线（如 `SetMode()`，不是 `Set_mode()`）
4. **C++ 成员变量必须 `m_` 前缀**，与局部变量和参数区分
5. **宏和枚举值必须全大写 + 前缀**，不允许无前缀的宏名
6. **文件名必须 snake_case**，C 和 C++ 统一

### 注释硬规则

7. **头文件中的对外接口必须有注释**（函数、类型、宏）
8. **协议格式 / 帧定义必须有注释**（字段含义、字节布局）
9. **硬件 workaround 必须注释原因**（不写清楚视为 bug）
10. **不允许无意义的注释**（如 `/* i++ */ i++;`）

## 11. 迁移策略

不建议一次性全部改名。建议按以下顺序逐步迁移：

### 第一批（影响最小）

- `class update_service` → `class UpdateService`
- `class version_store` → `class VersionStore`
- `class Led_service` → `class LedService`
- `Set_mode()` → `SetMode()`、`Get_mode()` → `GetMode()`

### 第二批（BSP 统一）

- `BSP_GPIO_*` → `BspGpio_*`
- `Bsp_UartRcv_*` → `BspUartRcv_*`
- `BspFlash_*` 保持不变

### 第三批（注释补全）

- 补全所有头文件中的接口注释
- 补全协议定义处的注释
- 清理无意义注释

**每批迁移建议在一个独立 commit 中完成**，只改名不改逻辑，方便 review 和回滚。
