# XY7320 mOTA 串口移植与升级流程阶段规划

> 适用工程：`XY_bootloadr` + `XY7320`  
> 当前基础：Bootloader + APP 分区已完成，USART2 固定码握手升级链路已实机跑通，后续如需接入完整 YModem / APP.fpk 可在当前裸 bin 流程上继续扩展。

---

## 1. 当前工程状态

当前工程已经完成：

```text
Bootloader: 0x08000000 ~ 0x0800FFFF  64KB
APP:        0x08010000 ~ 0x080DFFFF  832KB
Config:     0x080E0000 ~ 0x080FFFFF  128KB
```

APP 工程链接脚本已经改到：

```ld
FLASH : ORIGIN = 0x08010000, LENGTH = 832K
```

APP 入口也已经设置：

```c
SCB->VTOR = 0x08010000;
__enable_irq();
```

Bootloader 侧已经接入：

```text
MOTA/Core
MOTA/Config
MOTA/Bsp
MOTA/Port
```

并且已经实现 APP 区 Flash 的基础接口：

```c
read()
write()
erase()
```

当前已跑通：

```text
1. APP 通过 USART2 接收版本帧并写入 A2 升级配置。
2. APP 返回 XYA1 后复位进入 Bootloader。
3. Bootloader 识别配置区升级请求并停留升级模式。
4. Bootloader 进入升级、头包处理完成、固件接收完成分别返回 XYB1 / XYB2 / XYB3。
5. XY7320Host 按固定码推进自动升级流程，日志能定位卡住阶段。
```

后续还没有完成完整 YModem + APP.fpk 升级流程；当前稳定链路是固定码握手 + 12 字节头包 + APP 裸 bin。

当前已补充的 Bootloader 启动判断：

```text
1. Bootloader 上电后通过 USART1 输出调试日志。
2. PB8 作为强制升级按键，GPIO 上拉输入，低电平表示按下。
3. 上电后等待约 10 秒，期间每 10 ms 扫描 PB8。
4. PB8 连续稳定低电平 50 ms 后认为按键有效，停留在 Bootloader 并开启 USART2 DMA + IDLE 接收。
5. 10 秒内未检测到按键时，检查 0x08010000 APP 是否有效。
6. APP 有效则跳转 APP；APP 无效则停留在 Bootloader 等待升级。
```

当前已经在按键进入升级模式之外，补齐 APP 版本帧触发、升级配置区、Bootloader 配置识别和上位机固定码自动升级状态机。详见 `docs/project-docs/串口升级固定码握手流程.md`。

# 2. 总体开发路线

建议不要直接完整接入 mOTA + YModem，而是按能力逐步验证：

```text
阶段 1：串口基础移植
    USART1 printf 重定向
    USART2 DMA + IDLE 接收
    ↓

阶段 2：无协议升级 APP
    上位机发送 APP.bin
    Bootloader 接收裸数据
    擦除 APP 区
    写入 0x08010000
    读回 / CRC 校验
    跳转 APP
    ↓

阶段 3：APP 命令触发进入 Bootloader
    APP 收到升级命令
    设置升级标志
    软件复位
    Bootloader 识别标志后停留升级
    ↓

阶段 4：完整 mOTA 移植
    接入 BSP_UART
    接入 data_transfer
    接入 protocol_parser / YModem
    接收 APP.fpk
    校验并写入 APP 区
    跑通完整升级流程
```

核心原则：

```text
串口问题、DMA 问题、Flash 写入问题、APP 跳转问题、YModem 协议问题不要混在一起调。
```

# 3. 阶段 1：串口基础移植

## 3.1 目标

阶段 1 只验证串口基础能力，不做升级。

目标：

```text
USART1 用作日志 printf 输出口
USART2 用作升级数据接收口
USART2 使用 DMA + IDLE 判断一帧数据结束
```

推荐分配：

```text
USART1：日志口
USART2：升级口
```

不要把日志和升级数据混在一个串口里，否则后续 YModem 接收时容易被日志干扰。

---

## 3.2 当前已有基础

当前 `XY_bootloadr/Core/Src/usart.c` 已经有：

```c
UART_HandleTypeDef huart1;
UART_HandleTypeDef huart2;
DMA_HandleTypeDef hdma_usart2_rx;
```

USART1：

```text
PA9  -> USART1_TX
PA10 -> USART1_RX
115200 8N1
```

USART2：

```text
PA2 -> USART2_TX
PA3 -> USART2_RX
115200 8N1
```

```text
DMA1_Stream5
Channel 4
DMA_PERIPH_TO_MEMORY
DMA_NORMAL
```

`stm32f4xx_it.c` 中已有 DMA 中断：

```c
void DMA1_Stream5_IRQHandler(void)
{
    HAL_DMA_IRQHandler(&hdma_usart2_rx);
}
```

但是目前还需要补：

```text
1. USART1 printf 重定向
2. USART2 IDLE 中断开启
3. USART2_IRQn NVIC 使能
4. USART2_IRQHandler
5. USART2 DMA 接收缓存与完成标志
```

## 3.3 USART1 printf 重定向

可以先使用最简单的 `_write()` 重定向方式：

```c
#include "usart.h"
#include <stdio.h>

int _write(int file, char *ptr, int len)
{
    HAL_UART_Transmit(&huart1, (uint8_t *)ptr, len, 1000);
    return len;
}
```

验证：

```c
printf("bootloader start\r\n");
```

串口工具能看到日志，则 USART1 日志口通过。

## 3.4 USART2 DMA + IDLE 接收

建议先定义接收缓存：

```c
#define UART2_RX_BUF_SIZE 1024

uint8_t uart2_rx_buf[UART2_RX_BUF_SIZE];
volatile uint16_t uart2_rx_len = 0;
volatile uint8_t uart2_rx_done = 0;
```

启动接收：

```c
HAL_UART_Receive_DMA(&huart2, uart2_rx_buf, UART2_RX_BUF_SIZE);
__HAL_UART_ENABLE_IT(&huart2, UART_IT_IDLE);
```

USART2 中断处理：

```c
void USART2_IRQHandler(void)
{
    if (__HAL_UART_GET_FLAG(&huart2, UART_FLAG_IDLE) != RESET)
    {
        __HAL_UART_CLEAR_IDLEFLAG(&huart2);

        HAL_UART_DMAStop(&huart2);

        uart2_rx_len = UART2_RX_BUF_SIZE - __HAL_DMA_GET_COUNTER(huart2.hdmarx);
        uart2_rx_done = 1;

        HAL_UART_Receive_DMA(&huart2, uart2_rx_buf, UART2_RX_BUF_SIZE);
    }

    HAL_UART_IRQHandler(&huart2);
}
```

主循环验证：

```c
if (uart2_rx_done)
{
    uart2_rx_done = 0;
    printf("USART2 RX len = %d\r\n", uart2_rx_len);
}
```

阶段 1 验收标准：

```text
1. Bootloader 上电后 USART1 能打印日志。
2. PC 通过 USART2 发送 hello。
3. USART1 打印 USART2 RX len = 5。
4. 多次发送数据，长度都正确。
```

# 4. 阶段 2：无协议升级 APP

## 4.1 目标

阶段 2 先不接 YModem，不接 APP.fpk。  
先验证最核心的升级链路：

```text
串口接收数据 -> 擦除 APP 区 -> 写入 APP 区 -> 校验 -> 跳转 APP
```

这一阶段建议发送 `APP.bin`。

---

## 4.2 不建议完全裸发 APP.bin

如果直接裸发 APP.bin，Bootloader 不知道：

```text
文件有多大？
什么时候结束？
有没有丢包？
是否接收完整？
```

所以即使叫“无协议升级”，也建议加一个最小头部。

---

## 4.3 推荐最小升级头

```c
typedef struct
{
    uint32_t magic;
    uint32_t app_size;
    uint32_t app_crc;
} simple_app_head_t;

#define SIMPLE_APP_MAGIC 0x41505055U
```

`0x41505055` 可理解为 `APPU`。

上位机发送流程：

```text
1. 发送 simple_app_head_t
2. 等待 Bootloader 回复 OK
3. 发送 APP.bin
4. Bootloader 写入 APP 区
5. Bootloader 校验 CRC
6. 成功后跳转 APP
```

Bootloader 处理流程：

```text
等待升级头
    ↓
检查 magic
    ↓
检查 app_size <= APP_PART_SIZE
    ↓
擦除 APP 区 Sector 4~11
    ↓
分包接收 APP.bin
    ↓
写入 0x08010000
    ↓
读回或 CRC 校验
    ↓
跳转 APP
```

---

## 4.4 Flash 写入注意事项

APP 写入地址：

```c
#define APP_START_ADDR 0x08010000U
```

APP 区结束地址：

```c
#define FLASH_END_ADDR 0x08100000U
```

只允许擦写：

```text
Sector 4 ~ Sector 11
```

禁止擦写：

```text
Sector 0 ~ Sector 3
```

也就是禁止操作：

```text
0x08000000 ~ 0x0800FFFF
```

当前 `flash_port_stm32f4.c` 已经有保护逻辑，后续无协议升级也必须保持这个保护。

---

## 4.5 APP 有效性检查

写入 APP 后，检查 APP 首地址：

```c
uint32_t stack = *(volatile uint32_t *)0x08010000;

if ((stack & 0x2FF00000) == 0x20000000)
{
    // APP 看起来有效
}
```

这与当前 mOTA 配置一致：

```c
#define FIRMWARE_HEAD_DATA       0x20000000
#define FIRMWARE_HEAD_DATA_MASK  0x2FF00000
```


---

## 5.2 当前 mOTA 设计

当前配置中有：

```c
#define USING_IS_NEED_UPDATE_PROJECT USING_APP_SET_FLAG_UPDATE
#define FIRMWARE_UPDATE_VAR_ADDR     0x20000000
```

mOTA 内部有升级标志：

```c
volatile uint64_t update_flag __attribute__((section(".noinit"), aligned(8)));
```

设计意图是 APP 和 Bootloader 通过 RAM 中的升级标志通信。

---

## 5.3 当前需要注意的风险

在 GCC 下：

```c
volatile uint64_t update_flag __attribute__((section(".noinit"), aligned(8)));
```

不一定真的固定在 `0x20000000`。

如果链接脚本没有明确 `.noinit` 的位置，那么 APP 和 Bootloader 可能读写的不是同一个地址。

---

## 5.4 推荐做法

推荐使用固定 RAM 地址，例如 RAM 末尾：

```c
#define UPDATE_FLAG_ADDR 0x2001FFF0U
#define FIRMWARE_UPDATE_MAGIC_WORD 0xAAAAAAAA55555555ULL
```

APP 收到升级命令后：

```c
*(volatile uint64_t *)UPDATE_FLAG_ADDR = FIRMWARE_UPDATE_MAGIC_WORD;
NVIC_SystemReset();
```

Bootloader 启动后读取：

```c
uint64_t flag = *(volatile uint64_t *)UPDATE_FLAG_ADDR;

if (flag == FIRMWARE_UPDATE_MAGIC_WORD)
{
    *(volatile uint64_t *)UPDATE_FLAG_ADDR = 0;
    // 停留 Bootloader，等待升级
}
else
{
    // 正常检查 APP 并跳转
}
```

注意：

```text
APP 和 Bootloader 必须使用同一个 UPDATE_FLAG_ADDR。
```

# 8. 当前已跑通的固定码握手升级方案

截至 2026-06-08，当前工程已经跑通 `APP -> Bootloader -> XY7320Host` 全链路串口升级。当前方案仍属于“最小头包 + APP.bin”的阶段性方案，尚未切换到完整 mOTA/YModem/fpk，但已经具备可实机使用和可定位故障的自动升级流程。

当前自动升级时序：

```text
Host -> APP        发送版本帧：XYVH + yyyyMMddHHmm + flag + XYVT
APP  -> Host       返回 XYA1
APP                写入升级配置并 NVIC_SystemReset()
Bootloader -> Host 返回 XYB1
Host -> Bootloader 发送 12 字节头包：magic + app_size + crc32
Bootloader -> Host 返回 XYB2
Host -> Bootloader 按包大小连续发送 APP bin
Bootloader -> Host 返回 XYB3
Bootloader         跳转 0x08010000 APP
```

固定码定义：

```text
XYA1  APP 已接收版本帧，即将写配置并复位
XYB1  Bootloader 已进入升级模式，USART2 已准备接收
XYB2  Bootloader 已接收头包并擦除 APP 区，正在等待 APP 数据
XYB3  Bootloader 已接收完 APP，校验有效，即将跳转 APP
```

已经验证：

```text
1. APP 可通过 USART2 接收版本帧。
2. APP 可写入升级配置并复位。
3. Bootloader 可识别升级配置并停留升级模式。
4. Bootloader 可通过 USART2 返回阶段固定码。
5. 上位机可按固定码推进自动升级状态机。
6. Bootloader 可接收头包和 APP bin，写入 0x08010000 后跳转 APP。
```

后续如果继续接完整 mOTA/YModem/fpk，建议保留固定码握手思想：每个大阶段都先返回明确状态，再由上位机推进下一步，避免串口、Flash、跳转和协议层问题混在一起。

详细记录见：

```text
docs/project-docs/串口升级固定码握手流程.md
```
