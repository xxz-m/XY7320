# 升级协议概述

## 升级体系总览

XY7320 项目的固件升级体系存在**两条路线**，分别对应不同的开发阶段和复杂度需求：

| 路线 | 协议 | 传输方式 | 适用阶段 | 说明 |
|---|---|---|---|---|
| **简化协议** | 自定义 12 字节头包 | UART + DMA | 当前已实现 | 最小化头部，直接接收 APP.bin，详见 [[简化升级协议]] |
| **完整 mOTA/YModem** | YModem-1K + fpk | UART + DMA + IDLE | 规划中 | 完整 mOTA 框架，接收 APP.fpk，详见 [[mOTA移植指南]] |

> [!tip] 核心原则
> 串口问题、DMA 问题、Flash 写入问题、APP 跳转问题、YModem 协议问题**不要混在一起调**。按阶段逐步验证，每阶段跑通后再进入下一阶段。

---

## 四阶段开发路线

整体开发按能力逐步验证，不建议直接完整接入 mOTA + YModem：

```text
阶段 1：串口基础移植
    USART1 printf 重定向（日志口）
    USART2 DMA + IDLE 接收（升级口）
    ↓

阶段 2：无协议升级 APP（简化协议）
    上位机发送 12 字节头包 + APP.bin
    Bootloader 接收裸数据
    擦除 APP 区 → 写入 0x08010000
    读回 / CRC 校验 → 跳转 APP
    ↓

阶段 3：APP 命令触发进入 Bootloader
    APP 收到升级命令
    设置升级标志 → 软件复位
    Bootloader 识别标志后停留升级模式
    ↓

阶段 4：完整 mOTA 移植
    接入 BSP_UART / BSP_Timer
    接入 data_transfer / protocol_parser / YModem
    接收 APP.fpk
    校验并写入 APP 区
    跑通完整升级流程
```

> [!note] 当前进度
> 阶段 1（串口基础移植）和阶段 2（简化协议升级）已完成。阶段 3（APP 触发）和阶段 4（完整 mOTA）尚未实现。

## 串口分配

Bootloader 使用两个串口，**日志和升级数据必须分开**，否则后续 YModem 接收时容易被日志干扰：

| 串口 | 引脚 | 用途 | 说明 |
|---|---|---|---|
| **USART1** | PA9(TX) / PA10(RX) | 日志输出口 | 115200 8N1，printf 重定向 |
| **USART2** | PA2(TX) / PA3(RX) | 升级数据接收口 | 115200 8N1，DMA + IDLE 接收 |

USART2 RX DMA 配置：

```text
DMA1_Stream5, Channel 4
DMA_PERIPH_TO_MEMORY
DMA_NORMAL
```

## mOTA 层级调用链

mOTA 的整体架构是一个分层调用链，从 PC 上位机到 STM32 内部 Flash：

```text
PC 上位机
   │
   │ 串口字节流（YModem-1K / 简化协议）
   ▼
UART 驱动层（BSP_UART）
   │  USART2 DMA + IDLE 接收
   ▼
Data Transfer 数据传输层
   │  帧管理、数据缓冲
   ▼
Protocol Parser / YModem 协议层
   │  协议解析、包头校验、EOT 处理
   ▼
Firmware Manage 固件管理层
   │  固件校验、分包写入、完整性验证
   ▼
BSP Flash 抽象层
   │  分区管理、Flash 操作接口
   ▼
STM32F4 Flash Port
   │  HAL_FLASH_Program / HAL_FLASHEx_Erase
   ▼
内部 Flash APP 分区（0x08010000 ~ 0x080FFFFF）
```

> [!info] 当前工程已实现的部分
> 阶段二已接通 **Firmware Manage → BSP Flash → STM32F4 Flash Port** 这条链路。串口层和协议层尚未接入。

完整模式下的调用链：

```text
USART2 DMA + IDLE
    ↓
BSP_UART
    ↓
data_transfer
    ↓
protocol_parser / YModem
    ↓
Bootloader_Port_HostDataProcess
    ↓
Bootloader_Loop 状态机
    ↓
FM_WriteFirmwareSubPackage
    ↓
BSP_Flash_Write / BSP_Flash_Erase
```

## 串口与 YModem 的关系

> [!important] 底层通道 vs 上层协议
> **串口是底层通道，YModem 是上层协议。** 两者是独立的，可以先接串口，不接 YModem。

建议的接入顺序：

```text
1. UART 能初始化
2. UART 能发送调试信息
3. UART 能接收字节
4. UART 能回显收到的数据
5. 把收到的数据交给 data_transfer
6. 最后接 protocol_parser / YModem
```

这种分层方式使得：
- 阶段 2 可以用简化协议直接通过串口传输 APP.bin，无需 YModem
- 阶段 4 才引入 YModem 协议栈，接收 APP.fpk

## 升级标志机制

升级标志用于 APP 和 Bootloader 之间的通信：APP 设置标志后软件复位，Bootloader 启动时读取标志决定是否进入升级模式。

### mOTA 原始设计

mOTA 内部定义了升级标志变量：

```c
volatile uint64_t update_flag __attribute__((section(".noinit"), aligned(8)));
```

相关配置：

```c
#define USING_IS_NEED_UPDATE_PROJECT USING_APP_SET_FLAG_UPDATE
#define FIRMWARE_UPDATE_VAR_ADDR     0x20000000
```

设计意图是 APP 和 Bootloader 通过 RAM 中的 `update_flag` 通信，软件复位后 `.noinit` 段不被清零。

### GCC 下的风险

> [!warning] 地址不确定性
> 在 GCC + CMake 工具链下，`__attribute__((section(".noinit")))` **不一定真的固定在 `0x20000000`**。如果链接脚本没有明确 `.noinit` 段的位置，APP 和 Bootloader 可能读写的不是同一个地址。

### 推荐方案：固定 RAM 地址

推荐使用 RAM 末尾的固定地址，确保 APP 和 Bootloader 读写同一位置：

```c
#define UPDATE_FLAG_ADDR               0x2001FFF0U
#define FIRMWARE_UPDATE_MAGIC_WORD     0xAAAAAAAA55555555ULL
```

**APP 侧**（收到升级命令后）：

```c
*(volatile uint64_t *)UPDATE_FLAG_ADDR = FIRMWARE_UPDATE_MAGIC_WORD;
NVIC_SystemReset();
```

**Bootloader 侧**（启动后读取）：

```c
uint64_t flag = *(volatile uint64_t *)UPDATE_FLAG_ADDR;

if (flag == FIRMWARE_UPDATE_MAGIC_WORD)
{
    *(volatile uint64_t *)UPDATE_FLAG_ADDR = 0;  // 清除标志
    // 停留 Bootloader，等待升级
}
else
{
    // 正常检查 APP 并跳转
}
```

> [!caution] 关键约束
> APP 和 Bootloader **必须使用同一个 `UPDATE_FLAG_ADDR`**。

> [!note] 第一版建议
> 第一版完整升级流程跑通前，建议优先用**主机命令触发升级**（按键进入升级模式）；等流程稳定后再考虑 `update_flag + .noinit` 方案。

### 已完成

```text
Bootloader: 0x08000000 ~ 0x0800FFFF  64KB（Sector 0~3）
APP:        0x08010000 ~ 0x080FFFFF  960KB（Sector 4~11）
```

- APP 工程链接脚本已改到 `FLASH : ORIGIN = 0x08010000, LENGTH = 960K`
- APP 入口已设置 `SCB->VTOR = 0x08010000; __enable_irq();`
- Bootloader 已接入 MOTA/Core、MOTA/Config、MOTA/Bsp、MOTA/Port

- 简化升级协议（12 字节头包 + APP.bin）已跑通
