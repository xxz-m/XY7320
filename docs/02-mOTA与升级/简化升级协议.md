# 简化升级协议

> [!info] 当前状态
> 本协议是项目当前已实现的升级方案，用于在完整 mOTA/YModem 接入之前提供可靠的串口升级能力。

## 协议格式

### 12 字节头包

上位机在发送 APP 固件数据之前，先发送一个 **12 字节的小端序头包**，告知 Bootloader 固件的大小和校验值：

```text
偏移   长度   字段        说明
──────────────────────────────────────────
0      4      magic       魔数 0x41505055（小端序，HEX 为 55 50 50 41）
4      4      app_size    APP bin 文件大小（字节）
8      4      crc32       APP bin 文件的 CRC32 校验值
```

```c
typedef struct
{
    uint32_t magic;      // 0x41505055，可理解为 "APPU"
    uint32_t app_size;   // APP bin 文件大小
    uint32_t app_crc;    // APP bin 文件 CRC32
} simple_app_head_t;

#define SIMPLE_APP_MAGIC 0x41505055U
```

> [!note] 小端序说明
> 所有字段均为**小端序**（Little-Endian）。例如 magic `0x41505055` 在串口上的实际字节序为 `55 50 50 41`。

上位机示例（Qt 上位机界面会自动计算并显示头包 HEX）：

```text
55 50 50 41 18 13 00 00 D8 22 25 85
```

## 升级时序

完整的升级流程如下：

```text
上位机                              Bootloader
  │                                     │
  │  ① 发送 12 字节头包                  │
  │ ──────────────────────────────────>  │
  │                                     │ 检查 magic == 0x41505055
  │                                     │ 检查 app_size <= APP_PART_SIZE
  │                                     │ 擦除 APP 区（Sector 4~11）
  │  ② 回复 "OK"                        │
  │  <─────────────────────────────────  │
  │                                     │
  │  ③ 按包大小分包发送 APP.bin 数据     │
  │ ──────────────────────────────────>  │ 分包接收，写入 0x08010000
  │ ──────────────────────────────────>  │
  │              ...                    │
  │ ──────────────────────────────────>  │
  │                                     │
  │                                     │ ④ 校验 CRC / APP 首地址
  │                                     │ 跳转 APP
  │                                     │
```

### Bootloader 处理流程

```text
等待升级头
    ↓
检查 magic == 0x41505055
    ↓
检查 app_size <= APP_PART_SIZE
    ↓
擦除 APP 区 Sector 4~11
    ↓
回复 "OK"
    ↓
分包接收 APP.bin 数据
    ↓
写入 0x08010000
    ↓
读回校验 / CRC 校验
    ↓
检查 APP 首地址有效性
    ↓
跳转 APP
```

## 状态机

Bootloader 升级过程使用以下状态机：

```text
WAIT_HEAD ──收到合法头包──> RECV_APP ──接收完成──> DONE
   │                          │
   │  magic 错误               │  写入失败 / CRC 不匹配
   │  size 超限                │
   ↓                          ↓
 ERROR                      ERROR
```

| 状态 | 说明 |
|---|---|
| **WAIT_HEAD** | 等待上位机发送 12 字节头包，检查 magic 和 app_size |
| **RECV_APP** | 头包校验通过，擦除 APP 区后分包接收固件数据并写入 Flash |
| **DONE** | 全部数据接收完成，CRC / 首地址校验通过，准备跳转 APP |
| **ERROR** | 头包 magic 错误、app_size 超限、写入失败或 CRC 不匹配 |

## Flash 写入细节

### 地址与分区

```c
#define APP_START_ADDR  0x08010000U   // APP 写入起始地址
#define FLASH_END_ADDR  0x08100000U   // Flash 结束地址
```

### 写入约束

- **4 字节对齐**：STM32F407 片内 Flash 按 32bit 写入，地址必须 4 字节对齐
- **写入前必须擦除**：Flash 只能把 bit 从 1 写成 0，从 0 变回 1 必须擦除整个 Sector
- **最后一包补齐**：如果最后一包数据不足 4 字节，需要补齐（填 0xFF）后再写入

```c
HAL_FLASH_Program(FLASH_TYPEPROGRAM_WORD, address, data32);
```

### APP 区保护

```text
只允许擦写：Sector 4 ~ Sector 11（0x08010000 ~ 0x080FFFFF）
禁止擦写：  Sector 0 ~ Sector 3 （0x08000000 ~ 0x0800FFFF，Bootloader 区）
```

> [!caution] 安全约束
> `flash_port_stm32f4.c` 中已有保护逻辑，禁止擦写 Bootloader 区。后续所有升级模式都必须保持这个保护。

### APP 有效性检查

写入 APP 后，检查 APP 首地址处的栈顶值是否落在 SRAM 区域：

```c
uint32_t stack = *(volatile uint32_t *)0x08010000;

if ((stack & 0x2FF00000) == 0x20000000)
{
    // APP 看起来有效
}
```

这与 mOTA 配置一致：

```c
#define FIRMWARE_HEAD_DATA       0x20000000
#define FIRMWARE_HEAD_DATA_MASK  0x2FF00000
```

## DMA 双 buffer 方案

### 问题背景

> [!warning] 单 buffer 覆盖问题
> 如果 USART2 DMA 接收缓冲和 Flash 写入处理共用同一个 buffer，主循环写 Flash 时 DMA 可能已经把下一包数据覆盖进来，导致 APP 首地址被写坏。

错误流程：

```text
USART2 DMA 收到一包 → IDLE 中断触发
→ 主循环开始用 uart2_rx_buf 写 Flash
→ 中断里马上重新启动 DMA
→ 下一包继续写入同一个 uart2_rx_buf
→ 主循环还没处理完，buffer 前部已经被覆盖
→ Flash 写入的数据不是当前包的原始内容
```

### 解决方案

使用**两级 buffer**，将 DMA 接收和业务处理彻底隔离：

```c
uint8_t uart2_rx_buf[UART2_RX_BUF_SIZE];     // DMA 接收专用
uint8_t uart2_proc_buf[UART2_RX_BUF_SIZE];   // 主循环业务处理专用
```

| Buffer | 用途 | 说明 |
|---|---|---|
| `uart2_rx_buf` | DMA 专用接收缓冲 | 只在中断中由 DMA 写入，不在主循环中直接访问 |
| `uart2_proc_buf` | 业务处理缓冲 | IDLE 中断中从 DMA buffer 快速复制过来，主循环慢慢处理 |

---

## IDLE 中断处理流程

USART2 IDLE 中断中只做**快速动作**，不做耗时的 Flash 操作：

```c
void USART2_IRQHandler(void)
{
    if (__HAL_UART_GET_FLAG(&huart2, UART_FLAG_IDLE) != RESET)
    {
        __HAL_UART_CLEAR_IDLEFLAG(&huart2);

        HAL_UART_DMAStop(&huart2);

        // 计算本帧接收长度
        uint16_t rx_len = UART2_RX_BUF_SIZE - __HAL_DMA_GET_COUNTER(huart2.hdmarx);

        // 快速复制到业务 buffer
        memcpy(uart2_proc_buf, uart2_rx_buf, rx_len);
        uart2_rx_len = rx_len;
        uart2_rx_done = 1;

        // 立即恢复 DMA 接收
        HAL_UART_Receive_DMA(&huart2, uart2_rx_buf, UART2_RX_BUF_SIZE);
    }

    HAL_UART_IRQHandler(&huart2);
}
```

主循环处理：

```c
if (uart2_rx_done)
{
    uart2_rx_done = 0;
    BSP_Printf("USART2 RX len = %d\r\n", uart2_rx_len);
    Simple_Update_Process(uart2_proc_buf, uart2_rx_len);
}
```

> [!tip] 核心思路
> DMA buffer 只给 DMA 用，IDLE 中断复制到处理 buffer，立即恢复 DMA 接收，主循环慢慢写 Flash。

## 溢出保护

如果主循环还没处理完上一包，下一包已经到来，需要溢出保护：

```c
volatile uint8_t uart2_rx_overflow = 0;
```

当 `uart2_rx_done` 仍为 1 时又来了新数据，设置溢出标志并打印：

```text
USART2 RX overflow, slow down sender
```

> [!note] 处理方式
> 这表示上位机发送太快，需要**增大包间隔**，或者后续升级为 ACK 握手机制（PC 发一包 → Bootloader 写入并校验 → 回 ACK → PC 再发下一包）。

## 上位机配合参数

Qt 上位机（[[XY7320Host上位机说明]]）需要配合以下参数：

| 参数 | 值 | 说明 |
|---|---|---|
| **波特率** | 115200 | USART2 默认波特率 |
| **数据格式** | 8N1 | 8 数据位，无校验，1 停止位 |
| **包大小** | 可配置（建议 256~1024 字节） | 每包发送的数据量 |
| **包间隔** | 可配置（建议 >= 10ms） | 两包之间的延时，防止溢出 |

> [!tip] 参数调节建议
> 如果出现 `USART2 RX overflow`，应增大包间隔或减小包大小。当前简化协议没有 ACK 握手，完全依赖包间隔控制发送节奏。
