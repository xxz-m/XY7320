# 系统架构设计

## 1. 文档目的

这份文档用于回答两个问题：

- 原来的上下位机项目，应该从什么角度去理解和拆解
- 为了后续按产品方式重构，这个项目到底需要理解到什么程度

这份文档不是讲"怎么马上写代码"，而是讲"怎么把原项目真正读懂"。

对当前这个项目来说，先读懂，再重构，比直接补功能更重要。

## 2. 当前项目本质

这个项目不是一个单独的串口收发程序，也不是单独的 STM32 采集程序，而是一套完整的上下位机产品链。

它至少包含下面四个子系统：

1. 下位机 App
2. 下位机 Bootloader
3. 上位机 MainApp
4. 上位机 Launcher / Updater

围绕这四个子系统，又有三条核心链路：

1. 业务数据链
2. 参数与控制链
3. 升级链

如果只盯着某一个 `.c` 文件或某一个页面，很容易觉得项目很碎；但如果站在系统角度看，它其实是一套标准的产品结构。

## 3. 应该怎样拆原项目

理解原项目时，不要按"文件夹里有哪些文件"去理解，而要按"系统角色"和"数据流"去理解。

建议固定从下面 5 个角度拆：

### 3.1 角色拆分

先看系统里有哪些角色：

- 设备侧采集执行者：下位机 App
- 设备侧升级执行者：下位机 Bootloader
- 用户交互入口：上位机 MainApp
- 上位机程序更新入口：Launcher / Updater
- 外部存储：W25Q16
- 外部数据源：GPS / 其他模块

### 3.2 功能拆分

再看每个角色各自负责什么：

#### 下位机 App

- 6 路 ADC DMA 扫描采集
- 3 个频段功率/驻波比换算
- 2 路输入捕获
- GPS 数据获取
- 私有协议上传
- 升级包接收与外部 Flash 写入

#### 下位机 Bootloader

- 升级标志检查
- 外部 Flash 读取
- 升级包校验
- APP 区擦写
- 固件搬运
- 升级失败保护
- 跳转 APP

#### 上位机 MainApp

- 串口通信
- 主页面数据显示
- 参数设置
- 校准参数保存
- 日志管理
- 升级页面

#### 上位机 Launcher / Updater

- 替换 MainApp
- 启动主程序
- 处理桌面端更新流程

### 3.3 数据流拆分

再看数据怎么流动。

#### 业务数据流

```text
ADC / 频率捕获 / GPS
    -> 下位机 App 采集
    -> 功率 / 驻波比换算
    -> 私有协议打包
    -> 串口上传
    -> 上位机 MainApp 接收解析
    -> 页面显示 / 日志记录
```

#### 参数控制流

```text
上位机设置页面
    -> 生成参数命令
    -> 串口发送
    -> 下位机 App 接收
    -> 执行命令 / 返回应答
    -> 上位机记录结果
```

#### 升级数据流

```text
U 盘升级包
    -> 上位机读取
    -> 发送给下位机 App
    -> 下位机 App 写入 W25Q16
    -> 设置升级标志并重启
    -> Bootloader 读取 W25Q16
    -> 校验 / 解包 / 搬运到 APP 区
    -> 成功跳转 APP
    -> 失败回退或停留在升级态
```

### 3.4 状态流拆分

理解原项目时，还要看系统"会切换哪些状态"，而不是只看函数调用。

至少要搞清楚下面这些状态：

#### 下位机 App 状态

- 上电初始化
- 正常采集
- 正常上传
- 接收升级命令
- 接收升级包
- 写外部 Flash
- 设置升级标志
- 软件复位

#### 下位机 Bootloader 状态

- 上电启动
- 检查是否需要升级
- 检查 APP 是否有效
- 进入升级处理
- 校验升级包
- 写 APP 分区
- 升级成功
- 升级失败
- 跳转 APP

#### 上位机 MainApp 状态

- 未连接设备
- 已连接设备
- 正常收数
- 参数设置
- 升级准备
- 升级传输
- 升级结束

### 3.5 模块依赖拆分

最后再去看模块之间的依赖关系。

比如下位机 App 至少可以拆成：

- `adc采集模块`
- `功率换算模块`
- `输入捕获模块`
- `gps模块`
- `串口协议模块`
- `升级接收模块`

上位机至少可以拆成：

- `串口通信模块`
- `协议解析模块`
- `实时显示模块`
- `设置模块`
- `校准模块`
- `日志模块`
- `升级模块`

只有做到这一步，你后面重构时才知道：

- 哪些模块可以直接移植
- 哪些模块需要重写
- 哪些模块只是"页面逻辑"
- 哪些模块是真正的系统核心

## 4. 需要理解到什么程度

这个问题非常关键。

答案不是"把所有代码逐行背下来"，而是要达到一种可以独立讲清、独立拆分、独立重构的程度。

我建议把理解深度分成 4 级。

## 4.1 第一层：知道它能干什么

这一层是最浅的理解。

表现是：

- 知道项目大概有哪些功能
- 知道有上下位机
- 知道能采集、显示、升级

这一层只能算"用过"或"参与过"，不算真正理解。

### 判断标准

如果别人问你：

- 为什么要分 Bootloader 和 App
- 升级包为什么先写 W25Q16
- ADC 和频率捕获的数据怎么组织

你答不出来，那说明还停留在这一层。

## 4.2 第二层：知道模块和流程

这一层开始进入工程理解。

表现是：

- 知道系统由哪些模块组成
- 知道每个模块负责什么
- 知道主要数据流和控制流
- 知道基本升级流程

这一层已经能做简单维护，但还不够支撑你把项目当产品重构。

### 判断标准

如果你可以画出：

- 上下位机关系图
- 业务数据流图
- 升级时序图

说明你至少到了这一层。

## 4.3 第三层：知道为什么这样设计

这一层才是真正开始"掌控项目"。

表现是：

- 知道为什么业务协议和升级协议分开
- 知道为什么升级包不直接写 APP 区
- 知道为什么 Bootloader 不能做太多业务逻辑
- 知道为什么 GPS、ADC、频率捕获不必强行统一采集时基
- 知道为什么上位机要分 MainApp 和 Launcher

这一层是你现在应该努力达到的核心目标。

### 判断标准

如果别人问你"为什么这么设计"，你不只是会说"原来就是这样写的"，而是能解释：

- 技术原因
- 风险原因
- 工程原因

那就说明你到这一层了。

## 4.4 第四层：知道如何重构和替换

这一层是产品化和高级面试阶段最有价值的理解。

表现是：

- 知道旧项目哪些模块可以直接移植
- 知道哪些模块应该重写
- 知道哪些模块先做最小闭环
- 知道哪些功能可以后置
- 知道如何重新组织目录和模块边界
- 知道如何给项目补文档、版本、日志和验收标准

达到这一层，项目基本就从"参与过"变成"你能主导设计"。

## 5. 你当前至少应该达到哪一层

对你现在这个目标来说，建议最低要求是：

- 核心系统链路达到第三层
- 关键模块尽量达到第四层

也就是说：

### 必须达到第三层的内容

- 下位机 App 采集上传主链
- 下位机 Bootloader 升级主链
- 上位机 MainApp 显示与设置主链
- 协议链路
- 外部 Flash 在升级中的角色

### 尽量达到第四层的内容

- Bootloader 和 App 的边界重构
- 上位机模块重组
- 升级链与 mOTA 收口思路
- 目录结构与模块规范

## 6. 不需要理解到什么程度

这里也要避免一个误区：不是所有东西都必须钻到最底层。

当前阶段，不要求你：

- 把旧项目每一行代码都背下来
- 把所有 HAL 内部实现都看透
- 一上来就把 RTOS 任务调度也并进来
- 一开始就把所有页面都做到最终形态

你需要的是"系统级理解"，不是"代码记忆比赛"。

## 7. 什么叫真正读懂这个项目

如果要用一句话定义"真正读懂"，我会这样说：

**你能够脱离旧代码目录，单独把系统结构、数据流、升级流、模块边界、设计原因和重构思路完整讲出来。**

这比"我看过很多文件"重要得多。

## 8. 判断自己是否读懂的自测清单

如果下面这些问题你大部分都能独立回答，就说明你理解已经比较到位了。

### 8.1 系统结构

1. 为什么下位机必须分 Bootloader 和 App？
2. 为什么上位机最好分 Launcher 和 MainApp？
3. W25Q16 在系统里到底承担什么角色？
4. 这个系统里真正的主通信协议是什么？
5. Modbus RTU 在这个项目里是主协议还是某个模块协议？

### 8.2 下位机 App

6. 6 路 ADC 是怎么采的？
7. 功率/驻波比换算为什么放主循环里？
8. 2 路输入捕获为什么用两个定时器？
9. GPS 数据要哪些字段？
10. 为什么 ADC、频率、GPS 不一定要统一采样节拍？

### 8.3 协议链

11. 正常业务数据帧长什么样？
12. 参数下发命令怎么走？
13. 设备应答怎么设计？
14. 版本号、设备编号、序列号为什么要保留？

### 8.4 升级链

15. 为什么升级包先发给 App，而不是直接让 Bootloader 负责收包？
16. 为什么先写外部 Flash，再由 Bootloader 搬运？
17. Bootloader 上电后如何判断是进升级还是进 APP？
18. 升级失败时如何保护设备不变砖？
19. 为什么业务协议和升级协议应该区分？

### 8.5 上位机

20. MainApp 的核心功能模块有哪些？
21. 哪些日志需要自动保存？
22. 校准为什么先做参数输入和保存即可？
23. 升级页面和 Launcher 的职责分别是什么？

### 8.6 重构思路

24. 哪些旧代码你准备直接移植？
25. 哪些核心模块你准备重写？
26. 为什么先做业务闭环，再做升级闭环？
27. 为什么当前阶段不优先上 RTOS？

## 9. 推荐的理解输出物

为了保证自己不是"感觉懂了"，建议把理解过程输出成几个实物。

### 9.1 必做输出

1. 系统结构图
2. 业务数据流图
3. 升级时序图
4. 模块清单
5. 旧模块到新架构映射表

### 9.2 进阶输出

1. Flash 分区设计
2. 协议命令分类表
3. Bootloader / App 边界清单
4. 上位机页面与模块关系表

## 10. 当前最合适的理解目标

结合你现在的实际情况，当前最合适的目标不是：

- 把所有代码重写一遍
- 马上做最复杂的产品功能

而是先做到下面这件事：

**把原项目读成一套你能自己讲、自己拆、自己重建的系统。**

具体到当前阶段，建议目标如下：

1. 能画出系统架构图
2. 能讲清三条主线
3. 能讲清 Bootloader 升级链
4. 能讲清 MainApp 的页面和模块边界
5. 能把旧模块映射到新项目结构
6. 能说出哪些先做，哪些后做，为什么

## 11. 总结

你当前不需要把项目理解成"我会用这些代码"，而是要把它理解成：

- 我知道这套系统由哪些部分组成
- 我知道它们之间怎么协作
- 我知道为什么这样设计
- 我知道如何把它重新组织成更规范的新项目

对这个项目来说，真正够用的理解深度不是"会改几个功能"，而是：

**至少达到核心链路第三层理解，关键模块尽量达到第四层理解。**

只有到这个程度，后面把它当产品来设计才会越来越稳。
