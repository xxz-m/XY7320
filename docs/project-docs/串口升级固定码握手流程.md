# 串口升级固定码握手流程

本文记录 2026-06-08 跑通的 `APP -> Bootloader -> XY7320Host` 全链路升级流程。该流程替代早期“上位机按固定延时盲发头包和固件”的做法，改为每个关键阶段由下位机通过 USART2 返回 4 字节固定码，上位机收到对应固定码后再进入下一步。

## 1. 升级链路总览

参与端：

```text
XY7320 APP
  - 正常业务程序
  - 通过 USART2 接收版本帧
  - 写入升级版本配置后复位

XY_bootloadr
  - 复位后读取升级配置
  - 停留在 Bootloader 升级模式
  - 接收头包和 APP bin 数据
  - 写入 0x08010000 后跳转 APP

XY7320Host
  - Qt6/QML 上位机
  - 发送版本帧、头包、固件数据
  - 按固定码推进自动升级状态机
```

最终自动升级时序：

```text
1. Host 发送版本帧 XYVH + yyyyMMddHHmm + flag + XYVT
2. APP 返回 XYA1，表示版本帧已接收并准备复位
3. Bootloader 启动并返回 XYB1，表示已进入升级模式
4. Host 发送 12 字节头包
5. Bootloader 擦除 APP 区后返回 XYB2，表示等待 APP 数据
6. Host 按包大小连续发送 APP bin
7. Bootloader 接收完成并校验 APP 有效后返回 XYB3
8. Host 日志提示升级完成，Bootloader 跳转 APP
```

## 2. 固定码定义

固定码全部为 4 字节 ASCII，便于串口抓包、日志检索和上位机状态机匹配。

```text
XYA1  APP 已接收版本帧，即将写配置并复位
XYB1  Bootloader 已进入升级模式，USART2 已准备接收
XYB2  Bootloader 已接收头包并擦除 APP 区，正在等待 APP 数据
XYB3  Bootloader 已接收完 APP，校验有效，即将跳转 APP
```

排障含义：

```text
卡在 XYA1：APP 没收到版本帧，或版本帧解析/配置写入失败
卡在 XYB1：APP 已复位但 Bootloader 没进入升级模式，重点查版本配置区和启动判断
卡在 XYB2：Bootloader 没正确收到头包，或擦除 APP 区失败
卡在 XYB3：固件数据发送、Flash 写入、APP 有效性检查或跳转前流程异常
```

## 3. 版本帧格式

版本帧由上位机发给 APP，用于触发 APP 写入升级配置并复位。

```text
head      4 字节  "XYVH"
version  12 字节 yyyyMMddHHmm
flag      1 字节  APP_VERSION_FLAG_NEED_DOWNLOAD
tail      4 字节  "XYVT"
```

APP 入口位于：

```text
XY7320/Core/Src/app_update_config.c
```

关键行为：

```c
ret = AppVersionConfig_UpdateA2(version, flag);
if (ret == 0 && flag == APP_VERSION_FLAG_NEED_DOWNLOAD)
{
    const uint8_t ack[] = APP_UPDATE_ACK_VERSION_FRAME;
    (void)HAL_UART_Transmit(&huart2, (uint8_t *)ack, sizeof(ack) - 1U, 100U);
    HAL_Delay(20U);
    NVIC_SystemReset();
}
```

说明：APP 只有在版本帧解析成功、A2 配置写入成功且 flag 为 `NEED_DOWNLOAD` 时，才会返回 `XYA1` 并复位。

## 4. Bootloader 启动判断

Bootloader 启动后先初始化外设和 `Simple_Update`，再判断是否应进入升级模式。

进入升级模式的来源：

```text
1. APP 版本配置区请求升级
2. PB8 上电按键进入升级
3. APP 无效时强制停留 Bootloader
```

关键行为位于：

```text
XY_bootloadr/Core/Src/main.c
```

进入升级模式后：

```c
HAL_UART_Receive_DMA(&huart2, uart2_rx_buf, UART2_RX_BUF_SIZE);
__HAL_UART_ENABLE_IT(&huart2, UART_IT_IDLE);
(void)HAL_UART_Transmit(&huart2, (uint8_t *)ack, sizeof(ack) - 1U, 100U);
```

说明：`XYB1` 只有在 Bootloader 已经启动 USART2 DMA + IDLE 接收后才发送，所以上位机收到 `XYB1` 后可以安全发送头包。

## 5. 头包格式

上位机发给 Bootloader 的头包为 12 字节，小端序：

```text
magic     4 字节 0x41505055，对应 ASCII "APPU"
app_size  4 字节 APP bin 文件大小
crc32     4 字节 APP bin CRC32
```

Bootloader 处理入口：

```text
XY_bootloadr/MOTA/Core/simple_update.c
```

收到头包后执行：

```text
1. 检查 magic
2. 检查 app_size 不为 0 且不超过 APP_PART_SIZE
3. 擦除 APP 分区
4. 状态切换为 SIMPLE_UPGRADE_RECV_APP
5. 返回 XYB2
```

说明：上位机收到 `XYB2` 后再发送固件数据，可以确认 Bootloader 已经完成擦除并处于接收 APP 数据状态。

## 6. 固件数据发送

上位机按当前设置分包发送 APP bin：

```text
默认包大小：1024 B
可选包大小：128 / 256 / 512 / 1024 B
默认包间隔：10 ms
可选包间隔：1 / 5 / 10 / 30 ms
```

Bootloader 写入时会对最后一包做 4 字节对齐补齐，实际接收进度按原始 `app_size` 统计。

固件接收完成后：

```text
1. 检查 APP 栈顶是否落在 SRAM 区
2. 检查 APP Reset_Handler 是否落在 APP Flash 区
3. 返回 XYB3
4. 延时 100 ms
5. 关闭中断和外设并跳转 APP
```

## 7. 上位机自动状态机

上位机核心实现：

```text
tools/XY7320Host/backend/FirmwareUploader.cpp
tools/XY7320Host/backend/FirmwareUploader.h
```

自动状态：

```text
Idle
WaitAppVersionAck   等 XYA1
WaitBootReady       等 XYB1
WaitHeaderAck       等 XYB2
SendingFirmware     发送 APP bin
WaitFinishAck       等 XYB3
```

超时策略：

```text
每个握手等待阶段默认 8000 ms 超时
超时日志会输出当前阶段和最后缓存的 RX hex
```

日志示例：

```text
[AUTO] TX version frame, wait XYA1(APP accepted version frame).
[AUTO] RX code XYA1 at WaitAppVersionAck(XYA1).
[AUTO] APP accepted version frame and reset. Wait XYB1 bootloader ready.
[AUTO] RX code XYB1 at WaitBootReady(XYB1).
[AUTO] Bootloader ready. TX header packet, wait XYB2.
[AUTO] RX code XYB2 at WaitHeaderAck(XYB2).
[AUTO] Bootloader accepted header and waits app data. Start firmware TX.
TX firmware 1024 / 58324
[AUTO] Firmware bytes sent, wait XYB3(bootloader verified and will jump).
[AUTO] RX code XYB3 at WaitFinishAck(XYB3).
[AUTO] Upgrade complete. Bootloader verified firmware and is jumping to APP.
```

## 8. 本轮验证结论

已实机跑通：

```text
APP 收版本帧 -> APP 复位 -> Bootloader 停留升级 -> 头包握手 -> 固件写入 -> Bootloader 跳转 APP -> APP 正常运行
```

后续如果扩展到完整 mOTA/YModem/fpk，可以保留固定码思想：每个大阶段先返回明确状态，再由上位机推进下一步，避免问题混在串口、Flash、跳转和协议层之间。
