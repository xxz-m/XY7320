# XY7320 ModeManager 与 TinyFSM 设计说明

> [!abstract] 结论先行
> 对于 `XY7320` 现在的方向，以及后续要持续迁移 `XY7310A-+W` 旧项目业务这一目标，**引入 `ModeManager + FSM` 是值得的**。
> 
> 它的价值不是“为了用库”，而是把原本就存在的“模式切换复杂度”收拢成**统一入口、统一状态、统一切换顺序、统一资源管理**。

## 一、当前问题本质是什么

`XY7320` 现在面对的不是单一任务流程，而是一个**多模式、互斥任务、由上位机命令驱动切换**的系统。

当前业务特征：

- 上位机通过串口发送命令
- 协议层解析命令后，要触发设备切换工作模式
- 某些业务任务是互斥的，同一时刻只能有一个模式生效
- 当前模式下的业务逻辑还需要持续运行，而不是切换一次就结束
- 后面还要把旧项目中的 `POWER / MOHMS / CALI / GSM` 等业务不断迁入

这类问题如果不用统一状态管理，常见结果就是：

- 协议层直接改模式
- 主循环里用大段 `if / else if / switch` 执行业务
- 模式切换时资源清理和初始化散落在多个位置
- 后期新增模式时，旧逻辑越来越难维护

这就是为什么现在需要引入 `ModeManager/FSM`。

## 二、为什么值得做 `ModeManager/FSM`

### 1. 统一当前模式的真相源

系统任何时刻只能有一个“当前主模式”。

如果这个状态同时被多个地方修改：

- `ProtocolService` 改一次
- 某个任务自己改一次
- 其他 Service 再改一次

那么后面排查问题时，很难判断当前模式到底是不是真的正确。

引入 `ModeManager` 后：

- 当前模式只保存一份
- 所有切换必须走一个入口
- 谁改模式、怎么改模式都可追踪

### 2. 统一切换流程

状态切换最怕的不是“切不过去”，而是“切过去了但旧资源没退干净”。

状态机方案强制切换路径为：

```text
exit 旧状态 -> 切换当前状态 -> entry 新状态
```

这样能保证：

- 离开旧模式时一定会执行清理
- 进入新模式时一定会执行初始化
- 切换顺序固定，不依赖开发者临时拼逻辑

### 3. 协议层与业务层解耦

协议层应该只负责：

- 收包
- 校验
- 解码命令
- 把命令映射成“切换请求”

不应该直接负责：

- 改模式变量
- 启动 ADC
- 停止采样
- 切换继电器
- 修改任务调度关系

引入 `ModeManager/FSM` 后，协议层只负责“发事件”，业务模式层负责“决定怎么切”和“切完做什么”。

### 4. 适合后续持续迁移旧项目业务

旧项目 `XY7310A-+W` 已经天然具有模式结构。

它在 `UserAPP/configurations.h` 中已经定义了：

- `WAIT_MODEL`
- `POWER_MODEL`
- `GPS_MODEL`
- `MOHMS_MODEL`
- `CALI_MODEL`
- `BOOT_MODEL`
- `BUZZER_MODEL`
- `POWER_MODEL_GSM`

而且旧项目的主循环本质上就是：

- 当前模式是什么
- 就执行哪个模式的业务分支

所以 `ModeManager/FSM` 不是额外增加复杂度，而是在为旧业务迁移建立**稳定挂载点**。

## 三、TinyFSM 在这里解决了什么问题

TinyFSM 的核心价值，不在于“用了一个第三方库”，而在于它把状态行为组织成固定结构。

### 1. 让状态行为按状态聚拢

不用 FSM 时，常见逻辑是这样的：

```cpp
if (currentMode == Idle) {
    if (cmd == SwitchA) { ... }
}
else if (currentMode == AdcA) {
    if (cmd == SwitchIdle) { ... }
    if (cmd == SwitchB) { ... }
}
```

状态一多，切换逻辑和业务逻辑会迅速散开。

TinyFSM 风格则是：

- `IdleState` 自己管 Idle 的进入、退出、事件响应
- `AdcTaskAState` 自己管 A 的进入、退出、事件响应
- `AdcTaskBState` 自己管 B 的进入、退出、事件响应

这叫**按状态收拢行为**。

### 2. 固化 `entry / exit / react` 三段职责

TinyFSM 最适合这个项目的一点，就是它和当前需求天然匹配：

- `entry()`：进入该模式时的一次性初始化
- `exit()`：离开该模式时的一次性清理
- `react(event)`：当前模式收到某个事件后怎么处理

这可以直接映射到实际业务：

- 进入 `PowerMode` 时启动功率计相关资源
- 退出 `PowerMode` 时关闭相关采样、恢复继电器、清理缓存
- 收到 `SwitchToMohms` 时从 `PowerMode` 切到 `MohmsMode`

### 3. 为后续错误态、切换中状态预留标准结构

当前第一版可能只有：

- `Idle`
- `AdcTaskA`
- `AdcTaskB`

但后面大概率会出现：

- 初始化失败
- 切换失败
- 等待硬件稳定
- 异步完成后再真正进入目标模式

TinyFSM 在这里的优势是：

- 很容易引入 `ErrorState`
- 很容易增加 `SwitchingState`
- 不需要推翻原结构，只是新增状态类和事件类型

## 四、旧项目是否适合这种拆分

答案是：**非常适合**。

### 旧项目当前的真实结构

旧项目实际上已经具备“隐式状态机”：

- `workStatus` 是当前模式变量
- 协议层根据命令修改 `workStatus`
- 主循环根据 `workStatus` 进入不同业务分支

也就是：

```text
协议命令 -> 修改 workStatus -> 主循环按 workStatus 跑业务
```

### 旧项目当前的问题

- 协议层既负责解析命令，又直接改模式，又直接触发业务配置
- 主循环里模式分支过大，业务逻辑耦合严重
- 模式切换没有统一 `entry/exit`
- 某些模式切换时资源启停位置分散

### 拆分后的映射关系

旧项目中的元素，可以自然映射成新架构：

- `uartProtocolCmd()` 中的模式命令判断 -> `ProtocolService` 中的事件分发
- `systemConfig.workStatus` -> `ModeManager` 中的当前模式
- `main.cpp` 中的大 `if/else` 模式分支 -> 各状态的 `Tick/run` 逻辑
- 原来分散的初始化/清理动作 -> 各状态的 `entry/exit`

也就是说，旧项目不是“不适合状态机”，而是**本来就在用状态机思想，只是写法还没收拢**。

## 五、推荐的职责边界

这是整个方案最关键的部分。

### 1. `ProtocolService` 的职责

只负责：

- 接收串口数据
- 协议解码
- 校验数据包合法性
- 把命令翻译成“切换请求”或“读数据请求”

不负责：

- 保存当前模式
- 直接修改模式变量
- 启动/关闭 ADC
- 启动/关闭具体业务任务
- 直接操作 `PowerMeter`、`MohmsMeter` 等业务对象的模式切换

### 2. `ModeManager` 的职责

只负责：

- 保存当前模式
- 对外提供统一切换入口
- 检查切换是否合法
- 执行 `exit -> 切换 -> entry`
- 周期驱动当前模式运行一步业务

不负责：

- 协议解析
- ADC 底层驱动
- 具体算法实现
- 报文编码

### 3. 状态类 / Usecase / Task 的职责

只负责：

- 某个模式下具体要做的业务
- 模式进入时必须做的一次性动作
- 模式退出时必须做的一次性清理
- 周期执行当前模式的一步逻辑

不负责：

- 串口收包和协议解码
- 决定全局当前模式真相源存储位置

## 六、这种方案解决了哪些工程问题

### 1. 解决“谁都能改模式”的问题

原来如果多个模块都能改 `workStatus`，状态会失控。

引入 `ModeManager` 后：

- 所有切换统一走 `RequestSwitch()`
- 模式修改路径唯一

### 2. 解决“切换时容易漏清理”的问题

例如从 `PowerMode` 切到 `MohmsMode`：

- 忘关 ADC
- 忘停某个定时器
- 忘清某个波形缓存

状态机通过 `exit()` 固化了这个清理位置。

### 3. 解决“协议层和业务层搅在一起”的问题

原项目中协议层直接调用：

- `setPowerMeterModel()`
- `setMohmsMeterModel()`
- 修改 `workStatus`

这会导致协议一变，业务层也要跟着改。

拆开后：

- 协议只发切换请求
- 业务层在状态中自行完成初始化

### 4. 解决“主循环越来越大”的问题

旧项目的主循环已经有大块按模式分支的逻辑。

如果后面继续加：

- 校准模式
- GSM 模式
- 错误恢复模式

单一主循环会迅速膨胀。

状态机的价值是把这些逻辑分散到各状态类里，让每个文件只关心自己的模式。

## 七、这种方案不解决什么问题

这个也必须讲清楚。

`ModeManager/FSM` 不是万能方案，它主要解决的是**模式与切换**。

它不直接解决：

- 协议实现质量问题
- ADC 采样算法问题
- DMA 数据完整性问题
- 校准算法正确性问题
- 复杂数值计算性能问题

也就是说：

- 它能让结构更清晰
- 但不能替你自动修复业务算法本身

## 八、可能出现的问题与注意点

### 1. 过早做得太重

如果当前只有 `Idle / A / B` 三个简单模式，一上来就设计成很多层、很多事件、很多状态，会让理解成本升高。

建议：

- 先按 TinyFSM 风格建结构
- 第一版只做最小状态集

### 2. 把所有业务细节都塞进状态机

状态机负责“模式管理”，不是负责“把所有逻辑都塞进去”。

例如：

- ADC 采样算法
- 功率计算公式
- 查表校准

这些依然应该留在独立业务类中，由状态调用。

### 3. 把切换和执行混在一起

`RequestSwitch()` 和 `Tick/run` 要分开。

- 切换时只做一次性动作
- 周期运行时只做当前状态的一步逻辑

否则后面状态切换和业务执行会互相污染。

### 4. 异步切换场景需要后续演进

当前先用打印代替业务时，切换看起来是同步完成的。

但以后如果切换涉及：

- 停止硬件
- 等待硬件稳定
- 异步等待初始化成功

就会出现“切换中”状态。

这时需要：

- `SwitchingState`
- 或 `isSwitching` 标志
- 或 `ErrorState`

这也是 TinyFSM 的长期价值所在。

## 九、当前阶段最推荐的落地方式

### 第一阶段目标

先不要急着把旧项目整坨迁进来，也不要先接复杂 ADC 逻辑。

第一阶段只完成：

- 上位机命令可以切换模式
- 模式切换遵守 `exit -> entry`
- 当前模式能周期执行自己的打印逻辑

### 第一阶段最小状态集

建议先做：

- `Idle`
- `AdcTaskA`
- `AdcTaskB`

这是验证架构最小、最直观的一步。

### 第一阶段最小边界

- `ProtocolService`：把 `cmd` 转成切换事件
- `ModeManager`：做唯一切换入口
- `State`：打印 `entry/exit/tick`

### 第二阶段

把旧项目的真实模式开始迁进来，例如：

- `WaitMode`
- `PowerMode`
- `MohmsMode`
- `CaliMode`
- `PowerGsmMode`

### 第三阶段

再把真实业务逻辑逐步替换进去：

- `entry()` 放模式初始化
- `exit()` 放模式清理
- `Tick` 或 `RunStep()` 调用真实业务对象

## 十、最终建议

> [!success] 最终建议
> 对 `XY7320` 来说，先做 `ModeManager/FSM` 是正确方向。
> 
> 它不是为了“设计得更高级”，而是为了让后续旧项目迁移有一个稳定骨架：
> 
> - 协议只负责发请求
> - 模式只由一个中心管理
> - 每个业务模式有自己的进入、退出和运行逻辑
> - 后续新增模式时，不再回到“大 if/else + 到处改状态”的老路

一句话总结：

**你的项目不是“要不要硬上状态机”，而是“原本已经是状态机问题，现在要不要把它收拢成清晰结构”。答案是：值得，而且应该尽早做。**

## 十一、后续实施建议

如果继续往下推进，建议按这个顺序实施：

1. 在 `App/mode_manager/` 建立 `ModeManager` 与状态文件骨架
2. 先接 `Idle / AdcTaskA / AdcTaskB` 三个演示状态
3. 让 `ProtocolService` 只转发切换命令
4. 验证打印链路无误后，再迁旧项目中的 `WAIT / POWER / MOHMS` 模式
5. 最后再接真实 ADC、继电器、功率计、兆欧表业务

> [!note]
> 先把“切换结构”搭对，再迁“业务内容”，迁移成本会最低，问题也最容易定位。
