# XY7320 任务详解

## 任务模型

XY7320 当前采用协作式 OS 调度器与 TinyFSM 状态机组合的任务模型。需要先区分两类对象：

- **OS 任务**：由 `OS_CreateTask()` 注册，由调度器根据优先级和延时状态调用任务入口函数。
- **FSM 状态**：由 `ModeManager` 持有状态指针，通过 `entry()`、`exit()`、`react()` 和 `tick()` 描述业务模式生命周期。

因此，`TaskStateDmr`、`TaskStateGsm` 和 GNSS 状态虽然名称中有 `TaskState`，但它们不是独立的 OS 任务；它们运行在 `Task_ModeManager` 所在的 OS 任务上下文中。

整体关系如下：

```text
OS_Start()
  └─ 协作式调度器
      ├─ Task_UpdateConfig()  ── ProtocolService::Update()
      ├─ Task_ModeManager()   ── ModeManager::Tick()
      │                           └─ currentState_->tick()
      │                               ├─ Idle
      │                               ├─ TaskStateDmr
      │                               ├─ TaskStateGsm
      │                               └─ TaskStateGnss
      └─ Task_LED()           ── LedService::Update()
```

`Task_ModeManager()` 每轮先驱动当前 FSM 状态，再处理输入捕获 Service，最后延时 1 ms 让出 CPU。捕获中断只记录原始边沿事件，周期和脉宽由 `InputCaptureService::Update()` 在任务上下文中计算。

## 任务优先级与周期

| OS 任务 | 优先级 | 周期或让出方式 | 主要职责 |
| --- | ---: | --- | --- |
| `Task_UpdateConfig` | 5 | 每轮 `OS_DelayMs(10)` | 串口接收、协议拼帧、命令分发、升级服务轮询 |
| `Task_ModeManager` | 3 | 每轮 `OS_DelayMs(1)` | 驱动 FSM 当前状态、处理输入捕获测量超时 |
| `Task_LED` | 1 | 每轮 `OS_DelayMs(10)` | 驱动 LED 服务状态机 |

优先级数值越大表示调度优先级越高。协作式调度器不会抢占正在运行的任务，因此每个任务必须快速返回并调用 `OS_DelayMs()` 或其他让出 CPU 的接口，不能在任务入口中执行无限循环。

## Task_UpdateConfig

### 入口

```cpp
extern "C" void Task_UpdateConfig(void *arg)
{
    (void)arg;
    ProtocolService::Instance().Update();
    OS_DelayMs(10);
}
```

### 处理链路

```text
BspUartRcv
  ↓
ProtocolService::Update()
  ↓
取出接收块
  ↓
拼接协议流
  ↓
查找帧头和完整帧
  ↓
校验帧
  ↓
区分升级命令与普通模式命令
  ↓
RequestSwitch() 或 UpdateService
```

普通模式命令由 `ProtocolService::HandleCommandPacket()` 分发：

- `WAIT_MODEL` → `SwitchToIdleEvent`
- `POWER_MODEL` → `SwitchToDmrEvent`
- `POWER_MODEL_GSM` → `SwitchToGsmEvent`
- `GPS_MODEL` → `SwitchToGnssEvent`

命令值统一来自 `Domain/protocol/configurations.h`，不在协议分发函数中直接写十六进制字面量。协议收到模式命令后，`ProtocolService` 只负责生成事件和发送应答，不直接调用状态的 `entry()` 或 `exit()`。

### 设计边界

`Task_UpdateConfig` 不负责：

- 直接修改 `ModeManager::currentState_`；
- 直接启动或停止 ADC；
- 直接解析 DMR/GSM 功率算法；
- 在串口任务中阻塞等待完整数据。

这样可以让串口接收、协议语义和业务状态切换保持分层。

## Task_ModeManager

### 入口

```cpp
static void Task_ModeManager(void *arg)
{
    (void)arg;
    ModeManager::Instance().Tick();
    InputCaptureService::Instance().Update();
    OS_DelayMs(1);
}
```

### 调度顺序

每次运行顺序固定为：

1. `ModeManager::Tick()` 调用当前状态的 `tick()`。
2. 当前状态执行本模式业务，例如等待周期脉宽或处理 ADC 数据。
3. `InputCaptureService::Update()` 消费 A/B 原始捕获事件队列。
4. Service 更新周期、脉宽、有效标志和超时标志。
5. 任务延时 1 ms，让出 CPU。

由于当前状态在输入捕获 Service 之前运行，状态本轮读取到的是上一轮已经计算完成的捕获结果。下一轮状态会读取最新结果。这种顺序避免在定时器中断中执行复杂业务计算。

### ModeManager 的切换过程

```text
ProtocolService::HandleCommandPacket()
  ↓
ModeManager::RequestSwitch(event)
  ↓
根据 event.type 选择 nextState 和 nextMode
  ↓
currentState_->exit()
  ↓
currentState_ = nextState
currentMode_ = nextMode
  ↓
currentState_->entry()
  ↓
后续每 1 ms 调用 currentState_->tick()
```

状态切换由 `ModeManager` 统一完成。协议层、升级层或其他业务层不能直接操作状态指针。

## TaskStateIdle

`TaskStateIdle` 是默认状态，也是停止业务采集后的安全状态。

### entry

进入 Idle 时记录进入日志。当前不启动 ADC，不启动特定模式的功率算法。

### tick

当前 `tick()` 为空。输入捕获硬件仍由 `InputCaptureService` 常驻更新，但 Idle 不读取 DMR/GSM 测量结果，也不启动功率 ADC。

### exit

退出 Idle 时记录日志，随后由目标业务状态负责自己的初始化。

## TaskStateDmr

### 硬件和业务对应关系

- 比较器输入：`PA15 / TIM2_CH1`。
- 输入捕获信号：`BSP_TIM_CAPTURE_SIGNAL_A`。
- ADC 采集入口：`AdcService::StartTaskA()`。
- Oscilloscope 模式：`SCOPE_MODE_400_450`。
- 测量内容：DMR 周期、脉宽、正向功率和反向功率。

### entry 阶段

进入 DMR 状态时执行：

1. 调用 `InputCaptureService::ResetSignal(BSP_TIM_CAPTURE_SIGNAL_A)`。
2. 清除 PA15 对应的旧捕获事件、旧周期和旧脉宽。
3. 调用 `AdcService::Stop()`，避免上一个模式的 ADC 继续运行。
4. 将 `m_powerCaptureStarted` 设为 `false`。
5. 进入“等待周期脉宽”阶段。

清除历史结果是必要的。否则上一次 DMR 状态或其他流程遗留的有效结果可能被当前状态误认为新的前置条件。

### tick 阶段一：等待周期和脉宽

当 `m_powerCaptureStarted == false` 时，状态读取：

```cpp
InputCaptureService::Instance().GetSignalA()
```

只有 `result.valid == true` 时才进入功率采集阶段。`valid` 同时依赖：

- 周期有效；
- 脉宽有效；
- 脉宽不大于周期；
- 没有超时。

无效或超时期间，DMR 状态保持等待，不启动 ADC，也不使用过期测量结果。

### tick 阶段二：采集正反功率

周期和脉宽有效后执行：

```cpp
AdcService::Instance().SetScopeMode(SCOPE_MODE_400_450);
AdcService::Instance().StartTaskA();
```

`StartTaskA()` 启动 DMR 的 ADC 采样配置，后续由 `AdcService::Update()` 消费 DMA 数据，完成正向和反向功率计算。

启动成功后设置：

```cpp
m_powerCaptureStarted = true;
```

下一轮 `tick()` 不再重复启动 ADC，而是持续调用：

```cpp
AdcService::Instance().Update();
```

### exit 阶段

离开 DMR 状态时调用 `AdcService::Stop()`，释放 ADC 采样占用并停止旧模式的功率结果更新，然后清除阶段标志。

## TaskStateGsm

### 硬件和业务对应关系

- 比较器输入：`PA2 / TIM5_CH3`。
- 输入捕获信号：`BSP_TIM_CAPTURE_SIGNAL_B`。
- ADC 采集入口：`AdcService::StartTaskB()`。
- Oscilloscope 模式：`SCOPE_MODE_GSM`。
- 测量内容：GSM 周期、脉宽、正向功率和反向功率。

### entry 阶段

进入 GSM 状态时执行：

1. 调用 `InputCaptureService::ResetSignal(BSP_TIM_CAPTURE_SIGNAL_B)`。
2. 清除 PA2 对应的旧捕获事件、旧周期和旧脉宽。
3. 调用 `AdcService::Stop()`。
4. 将 `m_powerCaptureStarted` 设为 `false`。
5. 进入“等待周期脉宽”阶段。

### tick 阶段一：等待周期和脉宽

状态读取：

```cpp
InputCaptureService::Instance().GetSignalB()
```

只有 GSM 的周期和脉宽结果有效，才允许启动 GSM 功率 ADC。这样 PA15 的 DMR 测量不会误触发 GSM 功率采集。

### tick 阶段二：采集正反功率

周期和脉宽有效后执行：

```cpp
AdcService::Instance().SetScopeMode(SCOPE_MODE_GSM);
AdcService::Instance().StartTaskB();
```

随后由 `AdcService::Update()` 处理 GSM ADC DMA 数据和功率测量结果。状态标志防止每次 `tick()` 重复调用 `StartTaskB()`。

### exit 阶段

离开 GSM 状态时停止 ADC 并清除功率采集阶段标志，避免 GSM 采集继续影响 Idle、DMR 或 GNSS 状态。

## GNSS 状态

GNSS 使用 FSM 状态承载业务生命周期，不创建独立 OS 任务。

### entry

进入 GNSS 状态时启动 `GnssService` 的 USART3 DMA+IDLE 接收。

### tick

由当前 `Task_ModeManager` 的 `tick()` 调用 GNSS 状态，再由 GNSS 状态调用：

```cpp
GnssService::Instance().Update();
```

`GnssService::Update()` 在任务上下文中完成接收块提取、NMEA 行重组和 GGA/RMC 解析，中断只负责接收数据和设置就绪状态。

### exit

退出 GNSS 状态时停止 USART3 接收，但保留 Service 配置，便于下一次进入时重新启动。

## Task_LED

`Task_LED` 是低优先级后台任务：

```text
Task_LED()
  ↓
LedService::Update()
  ↓
OS_DelayMs(10)
```

它只驱动 LED 服务状态机，不参与模式切换、协议解析、ADC 测量或输入捕获。LED 闪烁不能阻塞其他任务，也不能在其中加入高耗时日志或延时循环。

## Services 初始化顺序

`App_Main_Init()` 当前顺序反映了依赖关系：

1. `LogService::Init()`，保证后续初始化可以记录日志。
2. `LedService::Init()`。
3. `UpdateService::Init()`。
4. `ProtocolService::Init()`。
5. `ModeManager::Init()`，默认进入 Idle。
6. `AdcService::Init()`。
7. `GnssService::Init()`。
8. `InputCaptureService::Init()`。
9. `InputCaptureService::Start()`，启动 TIM2/TIM5 捕获。

输入捕获硬件当前常驻运行，状态层通过 `ResetSignal()` 清理对应 A/B 的业务历史；ADC 则由 DMR/GSM 状态按流程启动和停止。

## 常见误解

### TaskStateDmr 是否是一个 OS 任务

不是。它是 `fsm::State` 的派生类，由 `ModeManager` 持有并调用。

### DMR 和 GSM 是否同时启动 ADC

不是。DMR 和 GSM 是互斥的 FSM 模式，同一时刻只有当前状态可以启动 ADC。

### 周期脉宽是否由 OS 任务计时

不是。周期和脉宽时间戳来自 TIM2/TIM5 输入捕获寄存器。OS 任务只负责消费事件和进行 Service 层计算。

### 状态切换是否由状态自己完成

不是。状态通过 `react()` 保持轻量，统一切换入口是 `ModeManager::RequestSwitch()`。
