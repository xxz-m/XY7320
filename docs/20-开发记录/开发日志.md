## 2026-07-12：OS 任务与 DMR/GSM FSM 状态详解

### 背景

APP 已从通用 ADC A/B 状态收敛为 DMR/GSM 业务状态。代码中真正由 OS 注册的任务是 `Task_LED`、`Task_UpdateConfig` 和 `Task_ModeManager`；`TaskStateDmr`、`TaskStateGsm`、GNSS 状态由 `ModeManager` 驱动，不是独立 OS 任务。

### 文档更新

新增 [[30-工程资料/系统设计/XY7320任务详解]]，详细说明 OS 调度器、三个 OS 任务、Idle/DMR/GSM/GNSS 状态、协议任务、初始化顺序以及输入捕获到正反功率采集的业务流程。

DMR/GSM 状态统一采用：

```text
进入状态 → 清除对应 A/B 捕获历史 → 等待周期/脉宽有效
→ 启动对应正反功率 ADC → 持续更新功率 → 退出时停止 ADC
```

### 命名与决策同步

- 删除 `TaskStateAdcA/B`，使用 `TaskStateDmr/Gsm`。
- `MODE_ADC_TASK_A/B`、`EVT_SWITCH_TO_ADC_TASK_A/B` 和旧事件类改为 DMR/GSM 命名。
- 协议命令值统一来自 `configurations.h`，不在 `ProtocolService` 中硬编码。
- 输入捕获中断只记录原始边沿，周期、脉宽、有效性和超时由 Service 任务上下文处理。

## 2026-07-08：DMR 反向表（ADL8317）加入与 1600-p2v 转换

### 背景

DMR 测量四通道中 P2/P4 是反向功率检测（来自 ADL8317），运放配置是反相 + 同向偏移，导致原始 ADC 与功率呈"功率 ↑ → ADC ↓"的反向关系。直接把原始值送入 `Calibration_AdcToDbm` 会得到反单调的查表结果。

### 设计结论

在 `DMRPowerMeasurement::Update` 内部对 P2/P4 原始值做 `1600 - raw` 翻转后再查表。系数 1600 来自 ADC 满量程 3.3V / 12-bit ADC 范围内的中点近似。

### 本次代码改动

`Domain/measurement/dmr_power_measurement.cpp`：

- 新增 `ad_adl8317_mv_adc[123]` 与 `ad_adl8317_dbmX10[123]`，覆盖反向功率典型量程
- 新增 `Calibration_ADL8317AdcToDbm` 接口，复用 `Calibration_AdcToDbm` 插值逻辑
- `DMRPowerMeasurement::Update` 内部增加：
  ```cpp
  const uint16_t raw_p2v = p2v;
  const uint16_t raw_p4v = p4v;
  const uint16_t lookup_p2v = (p2v <= 1600u) ? (uint16_t)(1600u - p2v) : 0u;
  const uint16_t lookup_p4v = (p4v <= 1600u) ? (uint16_t)(1600u - p4v) : 0u;
  ```
- 当原始 ADC > 1600（运放饱和/异常）时按 0 查表，触发 fall-through，避免溢出

### 验证

- `cmake --build XY7320/cmake-build-debug-stm32` 重新构建通过。
- 解析最近 log 数据，正向通道计算结果与表插值一致；反向通道因实测硬件暂未完全到位，仅做静态代码核对（待 TBD-4 微调 1600 系数后做实测对照）。

### 后续

- 系数 1600 是否需要按 PCB 实际运放偏置微调 → 见 `校准与测量链路总览.md` 的 TBD-4。
- 反向功率（uW）上报需在协议层扩展一个对数域 → 线性域的换算。

## 2026-07-07：调试串口关闭后进入 System Bootloader 的硬件问题修复

### 背景

USART1 调试日志口连接串口助手后，点击关闭 COM8 会导致设备表现为“卡死”。使用官方串口助手也可复现，排除自研上位机关闭逻辑的特异问题。

### 现场结论

复现后不复位、不掉电，使用 ST-Link HotPlug 读取现场：

```text
VTOR = 0x1FFF0000
PC   = 0x1FFF36CC
LR   = 0x1FFF14E1
CFSR = 0x00000000
HFSR = 0x00000000
```

说明 MCU 已经进入 STM32F407 内置 System Memory ROM Bootloader，不是 APP HardFault，也不是项目自定义 `XY_bootloadr`。

### 根因

原理图中 CH340C 自动下载电路连接为：

```text
CH340C DTR# -> R38 -> Q4 -> RST
CH340C RTS# -> R39 -> Q3 -> BOOT0
```

关闭串口时控制线瞬态经自动下载电路影响 `RST/BOOT0`，导致复位启动采样进入 System Bootloader。

### 修复

硬件已去掉自动下载控制路径上的电阻：

```text
R38：断开 DTR# -> Q4 -> RST
R39：断开 RTS# -> Q3 -> BOOT0
```

保留 `BOOT0` 稳定下拉和 `RST/NRST` 正常复位电路。修复后关闭串口不再触发进入 `0x1FFF0000`。

详细记录见 [[20-开发记录/踩坑记录#2026-07-07：关闭调试串口后 STM32 进入 System Bootloader]]。

### 经验

- 调试口不应默认绑定自动下载电路，除非硬件提供跳帽/0R 可断开
- 遇到“像卡死”的问题，优先读取 `VTOR/PC/CFSR/HFSR` 判断 CPU 当前实际运行区域
- `0x1FFFxxxx` 表示 ST 内置 System Bootloader，不是项目 APP 或自定义 Bootloader

## 2026-06-26：ADC 采集 BSP/Service 闭环、上位机串口下拉框修复与日志降频

### 背景

基于 [[30-工程资料/STM32/ADC采样+DMA+TIM配置与流控方案]]，今天完成 ADC 采集第一版闭环实现，并修复上位机串口调试页端口下拉框不可选、ADC 平均值计算异常和运行一段时间后疑似卡死的问题。

### 一、ADC BSP 采样闭环落地

新增 `BSP/adc` 模块，职责限定为硬件采样和数据缓存，不感知 TaskA/TaskB 业务。

```text
XY7320/BSP/adc/bsp_adc.h
XY7320/BSP/adc/bsp_adc.c
```

当前采样链路：

```text
TIM3 TRGO → ADC1 8通道扫描 → DMA2_Stream0 Circular → s_dma_buf[512]
    → TC回调解交织 → s_samples[6][512] + s_idle_acc[2]
    → 第8轮TC置 ready 并停止 TIM3
    → App消费后 BspAdc_Resume() 继续下一批采样
```

关键接口：

```c
void BspAdc_Init(ADC_HandleTypeDef *hadc, TIM_HandleTypeDef *htim);
BspAdc_Status BspAdc_Start(void);
void BspAdc_Stop(void);
BspAdc_Status BspAdc_Resume(void);
bool BspAdc_IsDataReady(void);
const uint16_t *BspAdc_GetActiveSamples(uint8_t active_ch);
uint16_t BspAdc_GetIdleAverage(uint8_t idle_ch);
void BspAdc_OnConvCpltCallback(ADC_HandleTypeDef *hadc);
```

本次修正点：

- `scan` 循环变量必须从 0 初始化，避免随机越界读 DMA buffer
- `s_samples` 只有 6 路 active 通道，TC 回调中只保存 `BSP_ADC_ACTIVE_CHANNEL_COUNT`，CH6/CH7 只进入 idle 累加器
- `HAL_ADC_ConvCpltCallback()` 已接入 `BspAdc_OnConvCpltCallback()`
- `App_Main_Init()` 已初始化 `AdcService`，由其内部调用 `BspAdc_Init(&hadc1, &htim3)`

### 二、AdcService 接入 ModeManager 状态

新增 `Services/adc_service`，作为 App 状态机和 BSP ADC 之间的适配层。

```text
XY7320/Services/adc_service/adc_service.h
XY7320/Services/adc_service/adc_service.cpp
```

职责：

- `StartTaskA()` / `StartTaskB()`：进入对应模式时启动 ADC 采样
- `Update()`：轮询 `BspAdc_IsDataReady()`，ready 后处理当前模式数据
- `ProcessTaskA()`：读取 active CH0~CH3 的 512 点并求平均
- `ProcessTaskB()`：读取 active CH4~CH5 的 512 点并求平均，同时读取 Idle CH0~CH1 平均值
- 处理完调用 `BspAdc_Resume()` 恢复下一批采集

`TaskStateAdcA/B::tick()` 当前调用：

```cpp
AdcService::Instance().Update();
```

### 三、ADC 平均值计算问题修复

**问题现象：**

TaskA 平均值曾出现 `13,12,12,11`，TaskB 出现 `3,3`，而 Idle 通道约 `1000`。

**根因：**

`BSP_ADC_SAMPLE_A_COUNT/B_COUNT` 实际表示通道数，曾被误用为样本循环次数；随后只修改了除数，导致：

```text
TaskA：只累加 6 个点 / 512 → 约 12
TaskB：只累加 2 个点 / 512 → 约 3
Idle：BSP 内部真实累加 512 点 / 512 → 正常约 1000
```

**修复：**

TaskA/TaskB 的循环次数和除数全部统一为：

```cpp
BSP_ADC_TARGET_SAMPLE_COUNT  // 512
```

### 四、高频日志导致疑似死机问题缓解

**问题现象：**

进入 AdcTaskA/B 后运行一段时间疑似卡死。

**根因判断：**

ADC 每批约 5.12 ms ready 一次，而 `LOG_Printf()` 底层通过 `HAL_UART_Transmit()` 逐字符阻塞发送。每批都打印平均值会导致协作式调度器长期被串口阻塞，表现为系统卡死或响应明显变慢。

**修复：**

在 `AdcService::ProcessTaskA/B()` 中加入日志分频：

```cpp
if (++log_div >= 40u) {
    log_div = 0;
    LOG_Printf(...);
}
```

即约每 40 批打印一次，日志周期约 200 ms，ADC 采样仍保持原采样节奏。

> [!warning]
> 当前只是降频规避阻塞。后续如果需要长期高频观测数据，应将日志发送改为 UART DMA/环形队列异步输出，或通过协议批量上传数据。

### 五、上位机串口端口下拉框修复

**问题现象：**

- 串口调试页端口下拉框无法直接选择串口
- 固件页需要点开/关闭串口后端口号才刷新可选

**根因：**

- 调试页 `EDropdown` 直接绑定 `serialDebug.ports`，对 `QVariantList/QVariantMap` 访问不稳定
- `SerialDebug::refreshPorts()` 只在端口数量变化时发 `portsChanged()`，端口内容变化但数量不变时 QML 不刷新
- 调试页后端没有自动选中第一个可用端口

**修复：**

- `SerialDebugPage.qml` 改为和固件页一致，使用 `root.portModel()` 转换为 JS 对象数组
- `SerialDebug::refreshPorts()` 改为完整列表比较，内容变化也触发 `portsChanged()`
- Windows 下补充读取注册表 `HKEY_LOCAL_MACHINE\\HARDWARE\\DEVICEMAP\\SERIALCOMM`
- 当前端口为空或不存在时自动选中第一个可用端口；无端口时清空选择

### 四、UART2 DMA deinit 问题修复

**问题现象：**

APP 收到升级握手后复位，Bootloader 启动后无法发送 `XYB1`，上位机超时。

**根因分析：**

最近 git pull 在 `UpdateService::Init()` 里加了 UART2 DMA 初始化：

```cpp
BspUartRcv_Init(&huart2, s_dma_rx_buf, UART_DMA_BUF_SIZE);
BspUartRcv_Start();
```

但 `ResetToBootloaderAfterAck()` 复位前没有 deinit，导致：

- DMA 还在运行
- UART2 寄存器状态残留
- Bootloader 启动后 UART2 异常，发不出 `XYB1`

**修复方案：**

在 BSP 层新增 `BspUartRcv_DeInit()` 接口：

```c
void BspUartRcv_DeInit(void)
{
    HAL_UART_DMAStop(s_huart);
    __HAL_UART_DISABLE_IT(s_huart, UART_IT_IDLE);
    HAL_UART_DeInit(s_huart);
    s_huart = NULL;
    // ... 清空状态
}
```

`UpdateService::ResetToBootloaderAfterAck()` 复位前调用：

```cpp
void UpdateService::ResetToBootloaderAfterAck()
{
    BspUartRcv_DeInit();  // 复位前清理串口状态
    BspTimOs_DelayMs(50);
    BspTimOs_SystemReset();
}
```

这样 Bootloader 启动时 UART2 是干净状态，能正常发送 `XYB1`。

### 六、ADC 采集方案设计（TIM3 + ADC1 + DMA）

完成 ADC 采集方案设计与 CubeMX 配置，目标：8 通道扫描采集，TIM3 触发，DMA Circular 搬运，流控解耦采集与消费。

**通道分配：**

| 通道 | 归属 | 处理方式 |
|------|------|----------|
| CH0~CH3（IN0/1/4/5） | TaskA | 存 512 样本，后续接算法 |
| CH4~CH5（IN6/7） | TaskB | 存 512 样本 |
| CH6~CH7（IN8/9） | Idle | 累加 512 次取平均 |

**CubeMX 配置要点：**

- TIM3：PSC=0，ARR=839，TRGO=Update Event，NVIC 不使能（只需触发输出）
- ADC1：Scan 模式 Enable，8 通道，外部触发 TIM3_TRGO 上升沿，15 cycles 采样
- DMA2_Stream0：Circular 模式（**CubeMX 默认 Normal，必须手动改**），HalfWord，MemInc，FIFO Disable

**时序参数：**

- ADC 时钟 42 MHz，单通道 27 周期 = 0.643 µs
- 8 通道一轮扫描 = 5.14 µs，TIM3 触发周期 10 µs，余量充足
- DMA buffer 512 元素 / 8 通道 = 64 轮扫描填满一轮，TC 间隔 640 µs
- 8 轮 TC 凑满 512 样本/通道，总周期 5.12 ms

**流控方案：**

```text
TIM3 运行 → 64轮扫描 → TC中断 → 搬数据
    → 第1~7轮：TIM3不停，继续下一轮
    → 第8轮：置 adc_data_ready=1，HAL_TIM_Base_Stop(&htim3)
        ↓
App任务看到 flag → 处理数据 → 清 flag → BspAdc_Resume() → HAL_TIM_Base_Start(&htim3)
```

TC 回调中搬 64×6 = 384 次内存拷贝，耗时约 20~30 µs，远小于 DMA 回绕周期 640 µs，不会发生数据覆盖，因此第 1~7 轮不需要停 TIM3。

**启动顺序：** 先 `HAL_ADC_Start_DMA()` 再 `HAL_TIM_Base_Start()`，DMA 必须先挂上等 ADC 请求。

详细方案文档：[[30-工程资料/STM32/ADC采样+DMA+TIM配置与流控方案]]

HAL_UART_Transmit     检查 gState（位 [b0] 表示 TX）

Host -> Bootloader 发送 12 字节头包：magic + app_size + crc32
Bootloader -> Host 返回 XYB2
Host -> Bootloader 按包大小连续发送 APP bin
Bootloader -> Host 返回 XYB3
Bootloader         跳转 0x08010000 APP

### 二、Oscilloscope 算法理解文档化

在整理 `waveLimitFilter` 时发现两种常见误解：

- ❌ “先排序后丢掉小于平均的点”—— **错**。本算法不排序。
- ❌ “加权平均”—— **错**。所有点权重相同，是普通算术平均。

补两份配套文档：

```text
docs/30-工程资料/STM32/Oscilloscope算法说明.md   # 算法说明 + 适用场景 + 工程结论
docs/30-工程资料/STM32/waveLimitFilter动画.html # 浏览器内可交互动画
```

文档核心结论：

> 算法核心一句话：**斩波迭代 → 排序 → 截尾 → 算术平均**。

每一步对应代码里的具体函数。`waveLimitFilter` 与 `waveLimitFilter_x` 的区别仅在后者多了**抗闪烁保持（连续 10 帧零值才归零） + 跳变确认（大跳变需连续 2 帧才接受）**。

### 三、AdcService 接入 Oscilloscope 的实施方案

- 日志先打 Raw + Fil 两份做对比，确认滤波稳定后再切换协议上报路径。

### 四、`wavePEP6 = 0` 问题排查

实测 GSM 模式下日志出现 `AdcTaskB,,Fil,1043,0`，即 `wavePEP6_avg` 始终为 0。

定位结论：

- `Oscilloscope::initOscilloscope()` 中 `cfgGSM.Threshold6 = 1500` 默认值过高。
- `preprocessChannel` 把 `in[i] < 1500` 的点全部清零 → 进 `waveLimitFilter` 的数据已大部分为 0 → 第一轮斩波剩余 < 10 → `return 0`。
- **算法本身正确**，问题在 GSM CH6 阈值与前端放大倍数不匹配。

建议后续根据前端实测信号幅度单独配置 `Threshold6`。
