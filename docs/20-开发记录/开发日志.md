## 2026-06-15：升级模块拆分方案设计

完成升级模块从 `app_update_config.c` 单文件向四层分层的拆分方案设计。

详细方案见 [[30-工程资料/系统设计/升级模块拆分方案]]。

### 设计结论

当前 `app_update_config.c` 混合了 BSP（DMA 接收）、Services（流程编排）、存储（Flash 读写）三层职责。

拆分目标：

```text
Task_UpdateConfig → Update_service → Version_frame (Domain)
                                  → Version_store (Services) → Bsp_flash (BSP)
                                  → Bsp_uart_rcv (BSP)
```

新增 5 个模块共 10 个文件：

- `BSP/flash/bsp_flash.c/.h` — Flash 擦/读/写
- `BSP/uart/bsp_uart_rcv.c/.h` — 串口 DMA 接收
- `Domain/version_frame/version_frame.c/.h` — 版本帧协议解析
- `Services/version_store/version_store.cpp/.h` — A1/A2 版本配置存储
- `Services/update_service/update_service.cpp/.h` — 升级流程编排

修改 5 个文件，删除 4 个旧文件。

### 关键设计决策

1. `VersionFrame_Parse` 放入 Domain 层 — 它是纯协议解析，不依赖任何硬件
2. Flash 读写拆为 BSP + Services 两层 — BSP 只提供擦/读/写能力，Services 决定存什么、怎么组织
3. `Update_service` 作为胶水层 — 协调 Domain 解析、Services 存储、BSP 收发

### 下一步

按方案中的 12 步顺序逐步实施，每步编译验证。

## 2026-06-08：APP / Bootloader / XY7320Host 固定码握手升级全链路跑通

本次目标：

```text
1. 让 APP 收到上位机版本帧后返回固定码，并复位进入 Bootloader。
2. 让 Bootloader 进入升级模式、头包处理完成、固件接收完成三个阶段分别返回固定码。
3. 让 XY7320Host 自动升级流程不再依赖固定延时，而是按固定码推进状态机。
4. 让上位机日志能明确显示升级卡在哪个阶段。
```

### 固定码约定

```text
XYA1  APP 已接收版本帧，即将写配置并复位
XYB1  Bootloader 已进入升级模式，USART2 已准备接收
XYB2  Bootloader 已接收头包并擦除 APP 区，正在等待 APP 数据
XYB3  Bootloader 已接收完 APP，校验有效，即将跳转 APP
```

### 自动升级流程

```text
Host -> APP        发送版本帧：XYVH + yyyyMMddHHmm + flag + XYVT
APP  -> Host       返回 XYA1
APP                写入 A2 升级配置并 NVIC_SystemReset()
Bootloader -> Host 返回 XYB1
Host -> Bootloader 发送 12 字节头包：magic + app_size + crc32
Bootloader -> Host 返回 XYB2
Host -> Bootloader 按包大小连续发送 APP bin
Bootloader -> Host 返回 XYB3
Bootloader         跳转 0x08010000 APP
```

### 关键代码落点

APP 版本帧入口：

```text
XY7320/Core/Src/app_update_config.c
```

Bootloader 升级模式入口：

```text
XY_bootloadr/Core/Src/main.c
```

Bootloader 头包和固件接收：

```text
XY_bootloadr/MOTA/Core/simple_update.c
```

上位机自动握手状态机：

```text
tools/XY7320Host/backend/FirmwareUploader.cpp
tools/XY7320Host/backend/FirmwareUploader.h
```

### 验证结果

已实机跑通：

```text
APP 收版本帧 -> APP 复位 -> Bootloader 停留升级 -> 头包握手 -> 固件写入 -> Bootloader 跳转 APP -> APP 正常运行
```

Qt 上位机构建验证通过：

## 2026-06-08：Bootloader 按键进入升级模式与上位机下拉框修复

本次目标：

```text
1. Bootloader 上电后支持 PB8 按键进入升级模式。
2. 无按键时等待约 10 秒后自动跳转有效 APP。
3. 调试串口输出启动、等待、按键消抖、跳转和升级等待状态。
4. 修复 XY7320Host 发包设置下拉框显示不全、被遮挡和滚动条不明显的问题。
```

### 已完成事项

#### 1. Bootloader PB8 按键消抖

在 `XY_bootloadr/Core/Src/main.c` 中新增 PB8 按键检测参数：

```c
#define BOOT_KEY_GPIO_PORT       GPIOB
#define BOOT_KEY_GPIO_PIN        GPIO_PIN_8
#define BOOT_KEY_PRESSED_LEVEL   GPIO_PIN_RESET
#define BOOT_KEY_WAIT_TIMEOUT_MS 10000U
#define BOOT_KEY_SCAN_PERIOD_MS  10U
#define BOOT_KEY_DEBOUNCE_MS     50U
#define BOOT_KEY_LOG_PERIOD_MS   1000U
```

PB8 当前为 GPIO 上拉输入，低电平表示按下。Bootloader 启动后等待约 10 秒，期间每 10 ms 扫描一次，连续稳定低电平 50 ms 后认为按键有效。

#### 2. Bootloader 启动决策

当前启动判断流程：

```text
Bootloader 启动
  ↓
初始化 GPIO / DMA / USART1 / USART2
  ↓
USART1 输出调试日志
  ↓
等待 PB8 按键 10 秒
  ↓
按键有效：停留在 Bootloader，开启 USART2 DMA + IDLE 接收
  ↓
按键无效：检查 APP 是否有效
  ↓
APP 有效：跳转 APP
  ↓
APP 无效：停留在 Bootloader，开启 USART2 DMA + IDLE 接收
```

当前只预留"按键进入升级模式"的基础框架，后续 APP 写升级标志、上位机日期版本号、升级完成同步版本号等逻辑还未接入。

#### 3. 调试串口日志

启动和等待期间会输出：

```text

debug uart ready
upgrade key: PB8, pressed level: LOW
hold PB8 to enter upgrade mode, wait 10000 ms...
boot wait start, timeout = 10000 ms
boot wait: xxxx ms left, PB8=x
PB8 low detected, debounce...
PB8 debounce ok
boot wait timeout
APP valid, jump to APP
APP invalid, stay bootloader
waiting upgrade...
```

#### 4. 上位机下拉框显示修复

修改 `tools/XY7320Host/components/EDropdown.qml`：

```qml

```

滚动条支持常显：

```qml

active: root.scrollBarAlwaysVisible || contentListView.moving || contentListView.dragging
```

修改 `tools/XY7320Host/pages/UpgradePage.qml`：

```qml

popupDirection: 1
scrollBarAlwaysVisible: true
```

用于解决发包设置里的"包大小"和"间隔"下拉框在小窗口下显示不全、被下方控件遮挡、滚动条不明显的问题。

## 遇到并解决的问题

### user.h 缺失

`common.h` 包含 `user.h`，工程里没有，于是补了最小配置。

### RTOS 头文件误包含

因为未定义宏在 `#if` 中会被当成 0，导致误进入 RTOS 分支。

修复方式：

```c
#if defined(RTOS_USING_RTTHREAD) && (USING_RTOS_TYPE == RTOS_USING_RTTHREAD)
```

### GCC 不支持原始 update_flag 放置写法

补了 GCC 分支：

```c
volatile uint64_t update_flag __attribute__((section(".noinit"), aligned(8)));
```
