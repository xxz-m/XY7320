# 踩坑日志

## 2026-06-05：APP 从 Bootloader 跳转后卡死在 `__libc_init_array`

### 现象

Bootloader 已经成功接收 APP、写入 APP 分区，并且检查首地址通过：

```text
[simple update] app stack: 0x20020000
[simple update] app reset: 0x080106ED
[simple update] APP valid
[simple update] update ok, jump app
```

但是跳转后 LED 没有闪烁。

用 ST-LINK / OpenOCD halt 现场后看到 CPU 已经进入 APP 地址段，但处于 HardFault：

```text
current mode: Handler HardFault
pc: 0x08011250
msp: 0x2001ff90
```

`addr2line` / map 反查后，`0x08011250` 落在 newlib 的 `__register_exitproc` 附近，调用链大致是：

```text
Reset_Handler
  -> SystemInit
  -> copy data
  -> clear bss
  -> __libc_init_array
     -> __register_exitproc
        -> HardFault
```

### 为什么 Keil 能跑，CLion/GCC 不行

Keil/ARMCC 或 AC6 的启动文件、运行库、scatter 文件是一整套配好的：

```text
Reset_Handler
  -> SystemInit
  -> __main / __scatterload
  -> main
```

CLion + arm-none-eabi-gcc 使用的是 GCC/newlib 启动链路：

```text
Reset_Handler
  -> SystemInit
  -> copy data
  -> clear bss
  -> __libc_init_array
  -> main
```

如果 CMake 链接参数不够"裸机化"，newlib 可能拉入 `atexit`、`__register_exitproc`、锁对象、析构注册等 MCU 程序通常不需要的逻辑。MCU 固件一般不会从 `main()` 正常返回，也不需要 `exit()` 相关清理逻辑，这些东西反而可能在 bootloader 跳转场景里踩坑。

### 为什么第一版不注释也能跑

第一版大概率是因为 `.init_array` 基本为空，`__libc_init_array()` 进去转一圈就回到 `main()`。

后来工程启用了 CXX、工具链版本变化、链接参数变化，或者引入了更多运行库依赖，导致 `.init_array` 和 newlib 初始化内容变多，于是 `__libc_init_array()` 不再是"空转"，而是拉进了 `__register_exitproc` 等逻辑。

### 临时解决

如果 APP 是最小 C/HAL 程序，没有 C++ 全局对象、静态构造函数，可以临时注释启动文件里的：

```asm
bl __libc_init_array
```

这样可以绕过 newlib 初始化，直接进入 `main()`。

### 长期解决

维护 `XY7320/CMakeLists_template.txt`，让 CubeMX/CLion 重新生成 CMake 时也保留裸机 GCC 参数：

```cmake
add_compile_options($<$<COMPILE_LANGUAGE:CXX>:-fno-use-cxa-atexit>)
add_link_options(--specs=nano.specs --specs=nosys.specs -nostartfiles)
```

同时保留启动文件里的：

```asm
bl __libc_init_array
```

因为使用 `-nostartfiles` 后默认启动文件不再提供 `_init()` / `_fini()`，需要在 APP 工程里提供最小空实现：

```c
void _init(void)
{
}

void _fini(void)
{
}
```

这样既保留了 C/C++ 静态初始化入口，又避免拉入不适合当前裸机启动流程的默认启动对象。

### APP 侧仍需注意

APP 从 Bootloader 跳转启动时，`main()` 开头建议明确做：

```c
SCB->VTOR = 0x08010000U;
__enable_irq();
```

`SCB->VTOR` 保证中断向量表指向 APP 自己，`__enable_irq()` 用来恢复 Bootloader 跳转前关闭的全局中断。

不要在启动文件很早的位置提前开中断。那时 data/bss、HAL、外设状态可能还没准备好，反而更容易乱。

## 2026-06-05：USART2 DMA 接收包被覆盖，导致 APP 写入 Flash 错位

### 现象

上位机发送完整 APP，Bootloader 日志显示接收数量最终达到 APP 大小，但升级后 APP 首地址检查失败：

```text
[simple update] receive done
[simple update] app stack: 0x0801073D
[simple update] app reset: 0x0801073D
[simple update] APP invalid
```

本地 `XY7320.bin` 前 8 字节是正确的：

```text
00 00 02 20 ED 06 01 08
```

即：

```text
stack = 0x20020000
reset = 0x080106ED
```

用 ST-LINK 读板子 `0x08010000` 后发现 Flash 起始位置被写成了：

```text
3D 07 01 08 3D 07 01 08
```

`0x0801073D` 对应 APP 中的 `Default_Handler` 地址，说明 APP 前几个 word 被后续数据覆盖/错写。

### 根因

原来的逻辑让 USART2 DMA 接收和 Flash 写入共用同一个 buffer：

```c
uint8_t uart2_rx_buf[UART2_RX_BUF_SIZE];
```

流程变成：

```text
USART2 DMA 收到一包
IDLE 中断触发
主循环开始用 uart2_rx_buf 写 Flash
中断里马上重新启动 DMA
下一包继续写入同一个 uart2_rx_buf
主循环还没处理完，buffer 前部已经被覆盖
```

因此 Flash 写入的数据不是当前包的原始内容，而是被下一包 DMA 改写后的内容。

### 中间错误尝试

曾经把 `HAL_UART_Receive_DMA()` 从 IDLE 中断里挪到主循环写 Flash 之后。

这样虽然避免了 buffer 覆盖，但写 Flash 期间 DMA 没有接收，PC 继续发来的数据会丢失，表现为后半段接收不全：

```text
[simple update] recv 4521 / 4804
```

### 最终解决

使用"双 buffer"：

```c
uint8_t uart2_rx_buf[UART2_RX_BUF_SIZE];    // DMA 专用
uint8_t uart2_proc_buf[UART2_RX_BUF_SIZE];  // 主循环业务处理用
```

IDLE 中断里只做快速动作：

```c
HAL_UART_DMAStop(&huart2);

rx_len = UART2_RX_BUF_SIZE - __HAL_DMA_GET_COUNTER(huart2.hdmarx);

memcpy(uart2_proc_buf, uart2_rx_buf, rx_len);
uart2_rx_len = rx_len;
uart2_rx_done = 1;

HAL_UART_Receive_DMA(&huart2, uart2_rx_buf, UART2_RX_BUF_SIZE);
```

主循环只处理复制出来的 `uart2_proc_buf`：

```c
if (uart2_rx_done)
{
    uart2_rx_done = 0;
    BSP_Printf("USART2 RX len = %d\r\n", uart2_rx_len);
    Simple_Update_Process(uart2_proc_buf, uart2_rx_len);
}
```

这样 DMA 可以继续接收下一包，Flash 写入逻辑也不会再被 DMA 覆盖。

### 溢出提示

增加一个标志：

```c
volatile uint8_t uart2_rx_overflow = 0;
```

如果主循环还没处理完上一包，下一包已经来了，就打印：

```text
USART2 RX overflow, slow down sender
```

这表示上位机发送太快，需要增大包间隔，或者后续升级为 ACK 握手机制。

### 经验总结

不要让 DMA 接收和 Flash 写入共用同一个 buffer。

可靠的思路是：

```text
DMA buffer 只给 DMA 用
IDLE 中断复制到处理 buffer
立即恢复 DMA 接收
主循环慢慢写 Flash
```

如果后续要做正式升级协议，最好改成：

```text
PC 发一包
Bootloader 写入并校验
Bootloader 从升级串口回 ACK
PC 收到 ACK 后再发下一包
```

这样不再单纯依赖包间隔。
